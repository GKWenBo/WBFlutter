## 一、本课重点掌握（按重要程度排序）

### 1. `AsyncNotifier` + `keepAlive`：全局共享状态的标准形态 ⭐⭐⭐

```dart
@Riverpod(keepAlive: true)
class Cart extends _$Cart {
  @override
  Future<List<CartItem>> build() => ref.read(cartStorageProvider).load();  // 初始值异步读盘
  // ...加购/改量/删除
}
```

- 为什么用 **AsyncNotifier**（而非同步 Notifier）：初始值要从磁盘**异步**读取——和 M4 的
  ProductList 是同一种模式，只是"异步数据源"从网络换成本地存储。
- 为什么必须 **keepAlive: true**：购物车是"全局唯一、活满 App 生命周期"的状态，
  **不能靠"正好有人在 watch"这种偶然性兜底**。代码生成的 provider 默认 autoDispose——
  万一某个异步方法还没跑完、最后一个监听者恰好被移除，`state =` 会因 provider 已销毁而报错。
- 这个"AsyncNotifier + keepAlive"三件套，M8 登录态、M9 收藏、M10 订单**原样复用**。

### 2. 异步竞态：改状态前先 `await future` 拿最新值 ⭐⭐⭐（真实踩过的坑）

```dart
Future<void> addProduct(Product product, {int quantity = 1}) async {
  final current = await future;   // ← 关键：拿"当前已确定的最新列表"，不是某个旧快照
  // ...在 current 基础上算出 updated
  state = AsyncData(updated);
  _persist(updated);
}
```

- 坑长这样：如果"初始读盘"和"加购"各自独立地异步跑，**迟到的读盘结果可能用旧数据覆盖掉刚加购的新数据**。
- 解法：**每次修改前 `await future`**——它会等 `build()` 完成、且之后始终反映最新 state，不会用旧值覆盖新值。
- 判断要不要 `await future`：**看新状态是否依赖旧状态**。加购/改量是"在旧列表上改一部分"→ 必须先拿旧值；
  M8 的 login/logout 是"整个替换"（不管旧值都变成新用户/null）→ 不需要 `await future`。别无脑套模板。

### 3. 派生 provider：把"总件数/总价"算一次、多处复用 ⭐⭐⭐

```dart
@riverpod
int cartTotalCount(Ref ref) =>
    (ref.watch(cartProvider).asData?.value ?? const [])
        .fold(0, (s, item) => s + item.quantity);
```

- Tab 角标、结算页都要"总件数/总价"。做成**派生 provider**而不是各处自己 fold：
  ① 结果被缓存，多处 watch 只算一次；② 逻辑收在一处。
- **收窄重建的关键**（M13 详述）：Tab 角标 watch 的是派生出的 `int`，只有它按 `==` 真变了才重建——
  "改某行数量但总件数不变""结算页填地址"都不会惊动角标。
- `.asData?.value ?? const []`：loading/error 时先当空列表算，磁盘读一瞬间完成，用户几乎看不到"闪 0"。

### 4. 不可变模型 + `copyWith`：改数量 = 造新对象 ⭐⭐

```dart
CartItem copyWith({int? quantity}) => CartItem(..., quantity: quantity ?? this.quantity);
```

- Dart 对象默认不可变（字段 `final`），改数量靠 `copyWith` 造新副本，≈ Swift struct 的值语义修改。
- 列表更新也是造新列表（`[...current]` / `for-if` 推导），`state = AsyncData(新列表)`——
  **不是就地改**。Riverpod 靠"引用变了"感知更新；就地 mutate 旧列表不会触发重建。

### 5. 购物车行存"快照"，不存整个 Product ⭐⭐

- `CartItem` 只存加购那一刻的 `title/thumbnail/unitPrice/quantity` 快照，**不引用实时 Product**。
  即使商品后续改价/下架，购物车里已加入的行情不变——这是电商购物车的标准做法（≈ 下单快照表）。
- 呼应 M9 收藏、M10 订单：同样是"快照优于实时引用"的思路。

### 6. 持久化：JSON ⇄ shared_preferences ⭐⭐

```dart
Future<void> save(List<CartItem> items) async {
  final prefs = await SharedPreferences.getInstance();
  await prefs.setString(_key, jsonEncode(items.map((e) => e.toJson()).toList()));
}
```

- `shared_preferences` ≈ `UserDefaults`：只存基础类型。存"一组对象"要先 `jsonEncode` 成字符串——
  这就是 `CartItem` 要有 `@JsonSerializable()`（M2）的原因。
- 落盘是"背后悄悄做的副作用"：`_persist` **不 await、不阻塞 UI**（先更新 state 让 UI 立刻响应，磁盘慢慢写）。

## 二、新控件/API 速查表

| 概念/API | iOS 类比 | 怎么用 | 坑 |
|---|---|---|---|
| `AsyncNotifier`（`build` 返 Future） | 初始值异步的 ViewModel | `build()` 里 `await` 读盘/网络 | mutation 前按需 `await future` 防竞态 |
| `@Riverpod(keepAlive: true)` | 常驻单例 SessionManager | 全局唯一、活满生命周期的状态 | 忘了它 → autoDispose 误清 → `state=` 报 provider disposed |
| `await future`（Notifier 内） | 读"当前已确定值" | 依赖旧状态的 mutation 前调 | 只在"新值依赖旧值"时用；整替换的不用 |
| 派生 provider | computed property（但带缓存） | `ref.watch(cartProvider)` 再 `fold` | 用 `.asData?.value ?? const []` 兜 loading/error |
| `state = AsyncData([...])` | reload 数据源 | 造新列表赋值 | 就地改旧列表不触发重建（引用没变） |
| `copyWith` | struct 值语义修改 | 改一个字段造新对象 | 忘了它就地改 final 字段编译不过 |
| `shared_preferences` | `UserDefaults` | `getString`/`setString` + JSON 编解码 | 只存基础类型；敏感数据（token）不能存这里（M8 用 secure storage） |
| `SnackBar` / `ScaffoldMessenger` | 顶部/底部临时提示 | `ScaffoldMessenger.of(context).showSnackBar(...)` | 加购成功反馈常用它；别在 build 里直接调（放回调） |
