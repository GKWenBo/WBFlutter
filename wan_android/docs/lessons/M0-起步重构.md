# M0 起步重构（4-Tab 骨架 + 目录规范）

> 本课产出：把旧的 3-Tab 占位工程改造成电商 App「WanShop」的 4-Tab 骨架
> （首页 / 分类 / 购物车 / 我的），并立好 feature-first 的目录规范。
> 这一课几乎不写业务，重点是**把"以后每个模块往哪放"这件事一次定清楚**——
> 定错了后面每加一个功能都要跟你打架。

---

## 一、本课重点掌握（按重要程度排序）

### 1. `main.dart` 保持极薄：入口只负责"启动" ⭐⭐⭐

```dart
void main() {
  runApp(const ProviderScope(child: WanShopApp()));
}
```

- `main()` ≈ iOS 的 `@main`/`AppDelegate`；`runApp()` ≈ 设置 `UIWindow.rootViewController` 并 `makeKeyAndVisible`。
- **装配逻辑（主题、路由、首页）全部搬进 `app/app.dart`**，`main.dart` 永远只有这几行。
  为什么：以后接 Riverpod（`ProviderScope`）、go_router（`MaterialApp.router`）时，改动点集中在一处，
  不会让入口文件越长越乱。iOS 里你也会把 `AppDelegate` 里的装配抽到单独的 `AppAssembly`。
- `ProviderScope` 是所有 Riverpod 状态的容器根（≈ 全局 DI 容器），必须包在最外层。
  M0 就先放上——即使这一课还没有任何 provider，先占好位比 M4 再回来改入口干净。

### 2. feature-first 分层：按"业务"切，不按"技术类型"切 ⭐⭐⭐

```
lib/
  main.dart            极薄入口
  app/                 全局装配（app.dart / main_scaffold.dart / router/）
  core/                跨 feature 的通用代码（network/ storage/ error/ widgets/）
  features/<name>/     每个业务模块自成一体
    data/ domain/ presentation/
```

- **不要**建 `lib/pages/`、`lib/models/`、`lib/services/` 这种"按技术类型"的目录。
  那种切法在项目大了以后，改一个"购物车"功能要横跨三四个顶层目录来回跳。
- feature-first：一个功能相关的页面/模型/数据源全在 `features/cart/` 一个目录下，
  ≈ 你在 iOS 里按模块（而非按 MVC 角色）组织工程时的思路。
- feature 之间**互不 import**。要共享的东西（网络、存储、错误模型、通用组件）下沉到 `core/`。

### 3. 四个 Tab 常驻：`IndexedStack` 保留各自滚动位置 ⭐⭐

- 底部 Tab 用 `IndexedStack`（而非 `if (index==0) HomePage() else ...`）：
  4 个页面同时存在于树里，只显示当前选中的那个，**其余的状态/滚动位置都保留**。
  ≈ `UITabBarController` 的默认行为（切 Tab 回来，列表还停在原来滚动的位置）。
- 坑：如果用条件渲染只保留当前页，切走再切回会**重建**，滚动位置和输入内容全丢。
- （M5 起，"当前第几个 Tab"改由 go_router 的 `StatefulShellRoute.indexedStack` 托管，
  底层还是 IndexedStack，但导航栈也一起管了——见 M5。M0 先用最朴素的 setState 版本理解概念。）

### 4. 包名不改、显示名改 ⭐

- 工程包名保持 `wan_android` 不动——重命名包名对学习无意义且极易踩签名/路径的坑。
- 只把 App 的**显示名/标题**改成 WanShop（`MaterialApp.title` + 各平台的 Info.plist/manifest）。
- 记忆钩子：**对外叫什么（显示名）和工程内部叫什么（包名）是两回事**，别混。

## 二、新控件速查表

| 控件/概念 | iOS 类比 | 怎么用 | 坑 |
|---|---|---|---|
| `MaterialApp` | `UIApplication` + 根 `UIWindow` + 全局 `UIAppearance` | App 根组件，装配 `title`/`theme`/`home`（M5 后换 `routerConfig`） | 整个树只该有一个；主题写这里，别在每个页面重复设 |
| `Scaffold` | `UIViewController` 的标准页面骨架 | 提供 `appBar`/`body`/`bottomNavigationBar`/`floatingActionButton` 插槽 | 一个页面一个 Scaffold；嵌套 Scaffold 会叠出两条 AppBar |
| `NavigationBar`（M3） | `UITabBar` | `destinations` 放各 Tab，`selectedIndex`+`onDestinationSelected` 管选中 | 这是 Material 3 的新控件；老代码里的 `BottomNavigationBar` 是 M2 版，别混用 |
| `IndexedStack` | `UITabBarController` 的页面常驻 | `index` 选中第几个，`children` 是全部页面 | 所有子页都会被 build（不是懒加载）；页面很重时首帧成本要留意 |
| `ProviderScope` | 全局 DI 容器根 | 包在 `runApp` 最外层 | 忘了包，任何 `ref.watch` 都会在运行时抛"找不到 ProviderScope" |

## 三、代码地图

```
lib/
  main.dart                  runApp(ProviderScope(WanShopApp()))
  app/
    app.dart                 WanShopApp：MaterialApp + 主题（M5 后升级成 .router）
    main_scaffold.dart       MainScaffold：底部 4-Tab（M5 后接 navigationShell）
  core/                      先建空目录/占位（network、storage、error、widgets 陆续填）
  features/
    products/  categories/  cart/  profile/   先各建 presentation 占位页
```

## 四、自测清单

1. 为什么 `main.dart` 只放 `runApp` 一行，而不把主题也写进去？
2. feature-first 和"按 pages/models/services 切"相比，改一个功能时各自要动几个顶层目录？
3. 用 `IndexedStack` 而不是条件渲染做 Tab，用户能直接感受到的差别是什么？

## 五、练习

把四个占位页各放一句不同的文案，`flutter run` 后切 Tab 确认：
① 四个 Tab 能正常切换；② 在"首页"里放一个长列表滚到中间，切到"我的"再切回来，
确认滚动位置**没丢**（这就是 IndexedStack 常驻的效果）。
