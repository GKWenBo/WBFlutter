# M9 收藏 + 我的 + 首屏离线缓存

> 本课产出：收藏功能（详情页♥一键切换、收藏页、"我的"页入口）+ 首屏离线缓存
> （断网也能打开 App 看到上次的商品，配一条"离线数据"横幅）。技术上没有全新框架，
> 而是**把 M7 的那套模式再用一遍**，重点辨析一个关键区别：**"数据真身" vs "数据副本"
> 的容错策略完全不同。**

---

## 一、本课重点掌握（按重要程度排序）

### 1. "数据真身" vs "数据副本"：容错策略要反过来 ⭐⭐⭐

同样是 `JSON ⇄ shared_preferences`，语义不同，读盘失败的处理就该相反：

| | 收藏 / 购物车（真身） | 首屏缓存（副本） |
|---|---|---|
| source of truth | **在本地**，丢了就真丢 | **在服务器**，本地随时可扔 |
| 读盘/解析失败 | 该谨慎（尽量别丢用户数据） | **直接当没有**（`return null`），绝不能拖垮首页 |
| iOS 类比 | Core Data 主存储 | NSCache / 磁盘缓存 |

```dart
// ProductsCache.load()：副本，坏了就当没有，绝不抛错拖垮首页
try { return list.map(Product.fromJson).toList(); }
catch (_) { return null; }   // App 升级后字段变了之类 → 当作无缓存
```

- 记忆钩子：**副本要"宽容"（坏了扔），真身要"珍惜"。** 分不清这个，要么丢用户数据、要么让一份
  可有可无的缓存把主页搞崩。

### 2. stale-while-error：宁给旧数据，不给白屏 ⭐⭐⭐

M4 埋的降级骨架，这一课兑现：

```dart
Future<List<Product>> build() async {
  try {
    final page = await repo.fetchProducts(...);
    cache.save(page.products);            // 成功顺手写缓存
    _isFromCache = false;
    return page.products;
  } on AppException {                      // ← 只捕获网络/服务器/解析类
    final cached = await cache.load();
    if (cached == null || cached.isEmpty) rethrow;  // 没缓存可兜 → 维持原错误态
    _isFromCache = true; _hasMore = false;           // 离线只有这一页
    return cached;
  }
}
```

- **只 `catch (AppException)`**，别的错误照常抛——catch 太宽会把真正的 bug 也吞成"离线态"，排查时极坑。
- `_isFromCache` 标志让 UI 亮一条 `MaterialBanner`"正在展示上次数据 + 重新加载"。
- 离线数据只有第一页，`_hasMore = false` 关掉上拉分页（别对着缓存去请求第二页）。

### 3. 收藏 = M7 模式的复刻 ⭐⭐

```dart
@Riverpod(keepAlive: true)
class Favorites extends _$Favorites {
  Future<List<Product>> build() => ref.read(favoritesStorageProvider).load();
  Future<void> toggle(Product product) async {
    final current = await future;                    // M7 教训 1：防竞态
    final updated = current.any((e)=>e.id==product.id)
        ? current.where((e)=>e.id!=product.id).toList()
        : [...current, product];
    state = AsyncData(updated);
    ref.read(favoritesStorageProvider).save(updated); // 不 await：后台副作用
  }
}
```

- **AsyncNotifier + keepAlive + `await future` 防竞态 + 落盘不 await**——M7 的四条纪律原样搬。
- `toggle` 一个入口管收藏/取消（已收藏就移除，否则追加），UI 侧一个♥按钮来回点（≈ `toggleFavorite(_:)`）。

### 4. 派生 + family：`isFavorite(productId)` 精准订阅 ⭐⭐

```dart
@riverpod
bool isFavorite(Ref ref, int productId) =>
    (ref.watch(favoritesProvider).asData?.value ?? const []).any((e) => e.id == productId);
```

- 详情页♥按钮 watch `isFavoriteProvider(product.id)`：收藏列表一变，**只有关心这个 id 的实例**重算并通知——
  比详情页直接 watch 整个列表再自己 `contains` 更精准，也把"怎么算已收藏"收拢到一处（UI 不散落 contains）。
- 呼应 M6 的 family、M7 的派生 provider、M13 的收窄重建——**同一套"精准订阅"思想的第三次应用**。

### 5. 收藏为什么不进 `_protectedPaths`（鉴权表）⭐

- 收藏是"**这台设备上的本地数据**"，不挂账号 → 未登录也能看，所以 `/favorites` **不在** M8 的鉴权表里。
  对比 `/profile` 展示的是账号数据，必须登录。
- 设计上留了活口：以后收藏若改成走服务端接口（挂账号），把 `'/favorites'` 加进 `_protectedPaths` 即可——
  又是"路径表 + 统一 redirect"的扩展性回报。

## 二、新控件/API 速查表

| 概念/API | iOS 类比 | 怎么用 | 坑 |
|---|---|---|---|
| "副本"缓存容错 | NSCache（可丢） | 读/解析失败 `return null` | 别用"真身"的谨慎逻辑对待副本，反之亦然 |
| stale-while-error | 离线兜底展示 | try 网络 / catch 读缓存 / 都无则 rethrow | **只 catch 业务异常**，别 catch 太宽吞掉真 bug |
| `MaterialBanner` | 顶部通告条 | 直接当 Widget 排进布局（跟内容滚），或 `showMaterialBanner` 悬浮 | 悬浮版要自己管移除时机；跟内容绑定的提示用内联版 |
| 派生+family bool | 精准的"是否…"查询 | `ref.watch(isFavoriteProvider(id))` | 列表变时只通知相关 id，比整表 watch+contains 省 |
| `IconButton`（♥切换） | toggle 按钮 | `icon` 按 isFavorite 切实心/空心 | 图标态由 provider 驱动，别自己在 State 里存一份（会不同步） |

## 五、练习

给 `ProductsCache` 加一个简单 TTL（存的时候带上时间戳，`load` 时超过 24h 就当过期返回 null）。
做完对比：收藏（真身）你**绝不会**给它加"过期自动删"，但缓存（副本）加 TTL 天经地义——
再次体会"真身 vs 副本"这条贯穿本课的分界线。
