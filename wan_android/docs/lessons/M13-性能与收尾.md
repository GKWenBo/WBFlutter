# M13 性能与收尾

> 收官课。不再堆功能，而是学会**看**性能（DevTools）、**调**性能（三把优化杠杆）、**守**质量（lint 收紧）。
> 面向 iOS 背景：Flutter 的渲染管线和你在 iOS 上调 `CADisplayLink` / 图层光栅化的直觉高度同构，只是术语不同。

---

## 0. 先建立心智模型：一帧是怎么画出来的

Flutter 每一帧要在 **~16ms（60fps）** 内跑完一条流水线，跨两个线程：

```
              UI 线程（Dart）                          Raster 线程（GPU 光栅化）
  ┌───────────────────────────────────┐      ┌──────────────────────────────┐
  │ build → layout → paint → 生成图层树 │  →   │ 把图层栅格化成像素、交给 GPU   │
  └───────────────────────────────────┘      └──────────────────────────────┘
        ↑ 你写的 Widget 代码在这跑                ↑ 图片解码/复杂裁剪/阴影在这重
```

- **UI 线程卡** = build/layout/paint 太重（如整棵树无谓重建）→ 优化方向：**少重建、少布局**。
- **Raster 线程卡** = 光栅化太重（大图解码、`Opacity`/`ClipPath`/阴影滥用）→ 优化方向：**RepaintBoundary 隔离脏区、给图片限尺寸**。
- 任一线程超 16ms → **掉帧（jank）**，界面卡顿。

> iOS 对照：UI 线程 ≈ 主线程跑 Auto Layout + `draw(_:)`；Raster 线程 ≈ Core Animation 把 CALayer 合成/光栅化交 GPU。"少重建"≈ 少触发 `setNeedsLayout`；"隔离脏区"≈ `shouldRasterize` / 独立 CALayer。

**性能优化的铁律：先测量，再优化。** 别凭感觉猜——打开 DevTools 看哪个线程、哪一帧、哪个 Widget 是元凶，再动手。

## 1. DevTools：Flutter 的 Instruments

`flutter run` 后终端会打印一个 DevTools 链接；或在 IDE 里点 "Open DevTools"。四个面板最常用：

| 面板 | 作用 | iOS 对照 |
|---|---|---|
| **Performance** | 帧耗时图（Frame Chart），红/黄柱=掉帧；点某帧看 UI/Raster 各花多久、火焰图定位到具体方法 | Instruments 的 Time Profiler / Core Animation |
| **Widget Inspector** | 点界面反查 Widget 树；看某节点的约束/尺寸/属性 | Xcode 的 View Hierarchy（Debug View Hierarchy） |
| **CPU Profiler** | Dart 方法级火焰图，找热点函数 | Time Profiler |
| **Memory** | 堆快照、对象数量、找泄漏（比如忘 dispose 的 controller） | Allocations / Leaks |

Inspector 右上角三个**开关（性能调试利器）**：

- **Highlight Repaints（重绘彩虹）**：每次重绘图层就换个边框颜色。**一片区域颜色狂闪 = 它在被反复重绘**，正是该加 `RepaintBoundary` 的地方。← 本课 Banner 优化就是靠它看出来的。
- **Track Widget Rebuilds**：统计每个 Widget 重建次数。**某个不该变的节点计数飙升 = 重建范围没收窄**。
- **Highlight Oversized Images**：把"解码尺寸远大于显示尺寸"的图片标红——提示你给 `Image` 加 `cacheWidth`/`cacheHeight` 省内存。

> 命令行也能开性能叠层：`flutter run --profile` 后按 `P` 显示 GPU/UI 两条耗时曲线（超过绿线就是掉帧）。**性能务必在 `--profile` 模式测，不要用 `--debug`**——debug 模式有断言/无编译优化，慢得不真实（≈ iOS 别拿 Debug 配置测性能，要用 Release）。

## 2. 三把优化杠杆（从轻到重）

### 杠杆一：`const` —— 让 Widget 根本不参与重建

`const` 构造出来的 Widget 是**编译期单例**，父级重建时 Flutter 发现它 `identical`，直接跳过它的 build/diff。这是**零成本、最高频**的优化。

```dart
const SizedBox(height: 12)          // ✅ 每次父重建都复用同一个实例
const Icon(Icons.star, size: 14)    // ✅
SizedBox(height: gap)               // ❌ gap 是变量就 const 不了，只能这样
```

本项目 M13 把 `prefer_const_constructors` 等规则设成**强制**（见第 4 节），编译器帮你揪出所有漏加 `const` 的地方。

### 杠杆二：收窄重建范围 —— 只让"变的那一小块"重建 ⭐ 本课核心

**问题**：`setState()` 会重建**整个** `build` 方法返回的子树，哪怕只有一个下标变了。

真实案例——首页 Banner（[home_banner.dart](../../lib/features/products/presentation/widgets/home_banner.dart)）。改造前：

```dart
int _current = 0;
// ...
onPageChanged: (i) => setState(() => _current = i),   // ❌ 翻一页 → 整个 build 重跑
// build 里同时含 PageView.builder 和圆点指示器
```

翻页时真正要变的只有**底部 3 个小圆点**，但 `setState` 把 `PageView.builder` 也一起重建了。打开 Track Widget Rebuilds 能看到 PageView 的计数每次翻页 +1（本不该）。

改造后——把"变化的值"抽成 `ValueNotifier`，只用 `ValueListenableBuilder` 订阅它：

```dart
final _current = ValueNotifier<int>(0);          // 变化的状态下沉成一个可监听值
// ...
onPageChanged: (i) => _current.value = i,        // ✅ 不 setState，build 不重跑
// 只有圆点这一行订阅它：
ValueListenableBuilder<int>(
  valueListenable: _current,
  builder: (context, current, _) => Row(...圆点...),   // ✅ 只有这个 builder 重跑
)
```

**重建半径从整个 Banner 缩到一行圆点**，PageView 彻底不参与。别忘了 `_current.dispose()`（≈ deinit 里注销订阅）。

> 这是一个通用范式，多种"局部刷新"工具同源：
> | 工具 | 适用 | iOS 对照 |
> |---|---|---|
> | `ValueListenableBuilder` | 单个可监听值的局部刷新（本课） | Combine 订阅单个 `@Published` |
> | `AnimatedBuilder` | 跟动画 controller 局部刷新 | `CADisplayLink` 回调里只更新一层 |
> | Riverpod `Consumer` 只包变化的子树 | 状态变化只重建该 Consumer | 只让某个 view 订阅 ViewModel |
> | `ref.watch(p.select((s) => s.field))` | 只在**某个字段**变时才重建 | `sink` 里 `removeDuplicates` + map |

`select` 尤其值钱：`ref.watch(cartProvider.select((c) => c.count))` 只在 `count` 变时重建，购物车里改地址/备注都不会惊动这个角标。

### 杠杆三：`RepaintBoundary` —— 隔离"频繁自绘"的图层

`const` 和收窄重建管的是 **UI 线程的 build**；`RepaintBoundary` 管的是 **Raster 线程的 paint**。它给子树切出**独立光栅化图层**，子树自己重绘时不会把兄弟节点拖下水（反之亦然）。

本课把整个 Banner 包了一层：

```dart
return RepaintBoundary(   // 圆点每 200ms 补间动画持续重绘，关在 Banner 内部
  child: AspectRatio(...Stack...),
);
```

⚠️ **反过度优化**：`RepaintBoundary` 不是越多越好——每个边界都要额外内存存图层、还有合成开销。**滥用反而更慢**。而且很多场景框架已经替你加了：

```dart
// home_page.dart：SliverChildBuilderDelegate 默认 addRepaintBoundaries: true，
// 已自动给每个格子套了 RepaintBoundary（ListView.builder / GridView.builder 同理）。
// 所以列表 cell 里【不需要】再手动包——重复包只会白增图层。
```

**判断准则**：只给"没走 builder 列表、又频繁独立重绘"的区域手动加（动画角标、进度条、Banner 这类）。加之前先用重绘彩虹确认它确实在狂闪。

## 3. 长列表性能清单（电商 App 的重灾区）

| 做法 | 为什么 | 反面 |
|---|---|---|
| 永远用 `.builder` 构造器 | 只 build 屏幕可见 + 少量预加载的 item，列表再长内存也恒定（懒加载） | `ListView(children: [...1000个])` 一次性全建，卡死 |
| cell 尽量 `const` / 轻量 | 快速滚动时 item 频繁 build，越轻越不掉帧 | cell 里做重计算/正则/JSON 解析 |
| 定高列表给 `itemExtent` | 框架跳过逐个测量，直接算滚动位置，滚动更省 | 高度不定又不给，滚动时反复 layout |
| 图片限解码尺寸 | `Image.network(url, cacheWidth: 300)` 或用 `cached_network_image`，避免把 2000px 大图解码进 100px 格子 | 原图直接进内存，Raster 线程 + 内存双爆 |
| 少用 `Opacity`/`ClipPath`/大 `BoxShadow` | 这些走 saveLayer，Raster 很贵 | 每个 cell 套半透明 + 复杂裁剪 |

本项目已落实前两条：商品卡走 `SliverChildBuilderDelegate`（懒加载 + 自动 RepaintBoundary），图片用 `CachedNetworkImage`（内存+磁盘双缓存，二次出现命中缓存不走网络）。

## 4. lint 收紧（工程化收尾）

M13 在 `flutter_lints` 之上又开了一批**性能 + 质量**规则（见 [analysis_options.yaml](../../analysis_options.yaml)）：

```yaml
prefer_const_constructors: true              # 漏 const 直接报（最关键）
prefer_const_constructors_in_immutables: true
prefer_const_declarations: true
prefer_const_literals_to_create_immutables: true
prefer_final_locals: true                    # 不重新赋值的局部变量强制 final
unnecessary_parenthesis: true
use_super_parameters: true
```

配套命令：

```bash
flutter analyze          # 看命中（提交前必跑，保持 0 issues）
dart fix --dry-run       # 预览能自动修哪些
dart fix --apply         # 一键批量修（const、super. 这类机械改动全自动）
```

> **选规则要贴合项目诉求**：本课**没开** `avoid_redundant_argument_values`。它会逼你删掉 `skip: 0`、`limit: 20` 这种"和默认值相同"的显式参数，但本教学项目**故意写明**这些参数来讲清语义——规则和目的冲突时，取舍站在项目一边，而不是无脑全开。这本身就是一课：lint 是为项目服务的，不是反过来。

iOS 对照：≈ 给工程接入 SwiftLint 并挑一组 opt-in 规则 / 调高 Xcode warning 级别，再用 autocorrect 批量修。

## 5. 本课控件 / API 速查表

| 控件 / API | iOS 类比 | 怎么用（关键点） | 坑 |
|---|---|---|---|
| `ValueNotifier<T>` | Combine `CurrentValueSubject` / 一个 `@Published` 值 | `final n = ValueNotifier(0); n.value = 1;` 变更即通知订阅者 | 必须 `dispose()`，否则泄漏；只按 `==` 判变，改集合内部不触发 |
| `ValueListenableBuilder` | 只订阅单值的局部 `body` | `valueListenable:` + `builder:`，只有 builder 重跑 | 别把不相关的重代码塞进 builder，失去"收窄"意义 |
| `RepaintBoundary` | `CALayer.shouldRasterize` / 独立图层 | 包住"频繁独立重绘"的子树，切独立光栅层 | 别滥用；列表 builder 已自动加，勿重复包 |
| `AnimatedBuilder` | `CADisplayLink` 回调里只刷一层 | 跟 `AnimationController` 做局部重建 | `child:` 参数缓存不动的子树，别每帧重建它 |
| `ref.watch(p.select(...))` | Combine `.map().removeDuplicates()` | 只订阅状态的某个字段，别的字段变不重建 | select 里别返回每次都新建的对象（会永远判"变了"） |
| `const` 构造 | 编译期常量对象 | 值全是编译期常量时加 `const`，重建直接复用 | 有一个运行期变量就加不了；`prefer_const_constructors` 会提醒 |
| DevTools Performance | Instruments Time Profiler | `--profile` 模式测；红黄柱=掉帧，点进去看 UI/Raster | 别用 `--debug` 测性能，数据失真 |

## 8. 练习（★ 已在本次代码里做成范例，可直接对照源码；其余留给你上手）

1. **抓一次真 jank**（需你亲自动手看）：`flutter run --profile`，打开 DevTools → Inspector → 开 **Highlight Repaints**，滑首页、翻 Banner，观察哪块在闪；确认 Banner 优化后圆点动画不再波及整屏。→ 详细步骤见下方「附：练习 1 操作步骤」。
2. ★ **收窄购物车角标**：本项目角标读的是**派生 provider** `cartTotalCountProvider`（见 [main_scaffold.dart](../../lib/app/main_scaffold.dart) 的注释），已经只在总件数变化时才重建——比 `.select` 更优（结果被缓存、多处复用）。注释里也讲清了「派生 provider vs `.select`」各自的适用场景。
3. ★ **给 Banner 图片限尺寸**：[home_banner.dart](../../lib/features/products/presentation/widgets/home_banner.dart) 已加 `cacheWidth = 显示宽度 × devicePixelRatio`，按需解码省内存。用 DevTools **Highlight Oversized Images** 对比：加之前会标红，加之后不再红。
4. ★ **`dart fix --dry-run`**：本项目跑出来是 `Nothing to fix!`——收紧的 7 条 lint 全项目零命中，说明 const 纪律已到位。你可以故意删掉某处 `const` 再跑，观察它如何被 `dart fix --apply` 自动补回。

### 附：练习 1 操作步骤（重绘彩虹）

1. `flutter run --profile -d <你的设备>`（性能务必 profile 模式；本机可用 Android 模拟器 `emulator-5554`）。
2. 终端打印 `The Flutter DevTools debugger and profiler ... available at: http://...`，点开链接。
3. 进 **Flutter Inspector** 面板，顶部工具栏点亮 **Highlight Repaints**（重绘彩虹）。
4. 回到 App：**翻 Banner** —— 只有底部圆点区域的边框在换色狂闪，上方搜索框、下方商品网格**不闪**（RepaintBoundary + 收窄重建双重生效）。
5. 再开 **Track Widget Rebuilds**：翻页时 `ValueListenableBuilder` 计数在涨，而 `PageView` / `HomePage` 计数**不涨**——直观验证"重建半径缩到一行圆点"。
6. 对照实验：把 `cacheWidth` 那行注释掉重跑，开 **Highlight Oversized Images**，Banner 图会标红；恢复后消失。
