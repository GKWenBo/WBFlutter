# M11 测试（mocktail / golden / integration_test）

> 前 10 课我们边写边测，已经攒了 24 个测试，但手法比较"土"：每种场景手写一个
> `_XxxRepository extends ...` 覆盖方法。本课不再堆同类测试，而是**补齐一个资深工程师
> 该有的测试工具箱**，并把你 iOS 的测试心智模型完整映射过来。

---

## 一、先建立坐标系：测试金字塔

| 层 | 是什么 | 本项目里的例子 | iOS 对照 | 特点 |
|---|---|---|---|---|
| **unit** | 测纯函数/单个类的逻辑，不渲染 UI | `cart_test` / `orders_test` / `products_notifier_mock_test` | XCTest 里测 ViewModel/Service | 最快、最多、最稳 |
| **widget** | 在内存里渲染 Widget 树后断言/点击 | `cart_flow_test` / `auth_flow_test` / golden | SwiftUI 的 ViewInspector / 快照测试 | 快、headless、覆盖单页交互 |
| **integration** | 启动**整个 App**，真机/模拟器上走主流程 | `integration_test/app_flow_test` | XCUITest | 慢、少而精、覆盖端到端旅程 |

**原则**：底层多而快，顶层少而精。别用一个慢吞吞的 integration_test 去测"金额算得对不对"——那是 unit 层一行断言的事。

## 二、本课重点掌握

### 1. mocktail：从"手写 fake"升级到"可打桩 + 可验证交互" ⭐⭐⭐

对比 `test/home_cache_test.dart` 里的手写 fake：

```dart
// 旧手法：每种返回值都要新写一个类，而且无法验证"谁调了它、传了什么参数"。
class _OkRepository extends ProductsRepository {
  @override
  Future<ProductListResponse> fetchProducts({int limit = 20, int skip = 0}) async { ... }
}
```

mocktail 的两大能力：

```dart
class MockProductsRepository extends Mock implements ProductsRepository {}
// 注意是 implements 不是 extends：不继承任何真实实现，没打桩就调用会报错

// ① 打桩（stub）：给"参数匹配 skip:0"的调用定制返回值
when(() => repo.fetchProducts(limit: any(named: 'limit'), skip: 0))
    .thenAnswer((_) async => firstPage);   // 异步返回用 thenAnswer
when(() => repo.fetchProducts(skip: 20)).thenThrow(NetworkException(...)); // 也能打桩抛错

// ② 验证交互（verify）：手写 fake 根本做不到的能力
verify(() => repo.fetchProducts(limit: 20, skip: 20)).called(1); // 恰好调用 1 次
verifyNever(() => repo.fetchProducts(skip: 40));                  // 从未调用
```

- iOS 对照：`when` ≈ stub 返回值；`verify` ≈ `OCMVerify` / 手写 mock 里数调用次数。
- 本课用它测了一个手写 fake 测不了的东西：**上拉翻页时 `skip` 传的是"已有条数"而不是写死的页码**——
  `verify(skip: 20).called(1)` 精确锁死了这个契约。这就是 verify 的价值。
- `any(named: 'limit')` = 参数通配符（我只关心 skip，limit 传啥都行）。
  若参数是**自定义类型**且要用 `any()`，得先 `registerFallbackValue(FakeXxx())`（本课参数都是 int，用不上）。

### 2. golden 测试 = Flutter 版快照测试 ⭐⭐

你 iOS 最熟的 snapshot testing 在这里叫 golden：把 Widget 渲染成 PNG，和事先存好的"基准图"逐像素比对。

```bash
# 首次 / 改了 UI 后，生成基准图：
flutter test --update-goldens test/golden/
# 之后普通跑，像素不一致就 fail（diff 图写到 failures/ 目录）：
flutter test test/golden/
```

```dart
await expectLater(find.byType(SectionHeader),
    matchesGoldenFile('goldens/section_header_with_more.png'));
```

- **坑一（最反直觉）**：测试环境默认用 **Ahem 占位字体**——每个字渲染成实心方块「█」。
  所以基准图里汉字全是方块，这是**刻意的**：避免不同机器字体差异让 golden 到处失败。
  你看到的 `section_header_with_more.png` 就是几个黑方块 + 右侧紫色方块（「查看全部」按钮）。
- **坑二**：基准图**必须提交进 git**，否则别人/CI 没有比对基准。
- **坑三**：跨平台像素可能有细微差异，团队通常固定在 CI 同一套环境生成 golden。
- iOS 对照：≈ FBSnapshotTestCase / swift-snapshot-testing，连"首次录制基准、之后回归比对"的工作流都一样。

### 3. integration_test：端到端，启动真 App 走主流程 ⭐⭐

```dart
void main() {
  IntegrationTestWidgetsFlutterBinding.ensureInitialized(); // 必须的第一行

  testWidgets('主流程：首页看到商品 → 点进详情 → 加入购物车成功', (tester) async {
    await tester.pumpWidget(ProviderScope(
      overrides: [ productsRepositoryProvider.overrideWithValue(mockRepo) ], // 只换数据来源
      child: const WanShopApp(),   // 启动真正的 App（含 go_router 全套路由）
    ));
    await tester.pumpAndSettle();
    await tester.tap(find.text('端到端测试商品')); // 真的走 go_router push 到详情
    ...
  });
}
```

和 widget 测试的三点关键区别：

| | widget 测试（`test/`） | integration_test（`integration_test/`） |
|---|---|---|
| binding | `AutomatedTestWidgetsFlutterBinding` | `IntegrationTestWidgetsFlutterBinding` |
| 跑在哪 | 内存里 headless | 可 `-d <设备>` 跑到**真机/模拟器** |
| 命令 | `flutter test` | `flutter test integration_test/ -d <设备>` |
| 测什么 | 单页交互 | 跨页面、跨 Provider 的完整旅程 |

- 只把最外层"数据来源"换成 mock，**其余全是生产代码**（路由、Provider、页面），
  所以它验证的是"这些东西串起来真的能通"。
- iOS 对照：≈ XCUITest 启动整个 App 点真实控件。

## 三、本课踩到的真实环境坑（重要）

`flutter test integration_test/app_flow_test.dart`（不指定设备、headless flutter-tester）
在本机会**卡住 7 分钟无输出**——真因是它去下载 **Flutter Web SDK**（integration_test 带了
webdriver 依赖），而本机代理下载大文件时 TLS 握手失败（记忆里那个老环境坑）。

**解法**：指定一个真实设备就绕开 web 路径：

```bash
flutter test integration_test/app_flow_test.dart -d emulator-5554
# → Gradle assembleDebug 打出 apk → 装到模拟器 → 跑通 → All tests passed
```

记忆钩子：**integration_test 本来就设计成跑在真实设备上**，headless 只是它的一个附带能力；
本机 web 工具链下不下来，直接上设备反而是正路。

## 六、练习

给 `products_notifier_mock_test.dart` 再加一个用例：**首屏加载失败但有缓存时，降级到缓存数据**。
提示：这次同时要 mock repository（`thenThrow` 断网）**和**预置一份 `ProductsCache`，
然后 `verify` 缓存的 `load()` 被调用、而 `fetchProducts` 只被调了一次（没重试）。
做完你会体会到：mock 能让你精确编排"网络失败 + 有缓存"这种多组件协作的边界场景。
