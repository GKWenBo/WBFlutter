# M6 分类 & 搜索（family / autoDispose / 防抖）

> 本课产出：分类页（全部分类 → 某分类商品列表）和搜索页（输入防抖 → 结果网格）。
> 技术重点是 Riverpod 的 **`family`（带参数的 provider）** 与 **`autoDispose` vs `keepAlive`
> 的缓存策略**，以及一个非常常用的交互模式——**输入防抖（debounce）**。

---

## 一、本课重点掌握（按重要程度排序）

### 1. `family`：带参数的 provider，按参数各自缓存 ⭐⭐⭐

```dart
@riverpod
Future<List<Product>> categoryProducts(Ref ref, String slug) async {
  final page = await ref.watch(productsRepositoryProvider).fetchByCategory(slug);
  return page.products;
}
// 用：ref.watch(categoryProductsProvider('smartphones'))
```

- 加一个参数就成了 family——**每个不同参数值是一个独立的 provider 实例**（各自的 loading/data/error、各自缓存）。
  ≈ 一个"按 key 建的 ViewModel 工厂"。手机分类和电脑分类互不干扰，切回来命中各自缓存。
- 搜索同理：`searchResultsProvider(query)` 按关键词各自缓存。

### 2. `autoDispose` vs `keepAlive`：什么该留、什么该清 ⭐⭐⭐

代码生成的 provider **默认就是 autoDispose**（`@riverpod`）：没人 watch 就回收。
要常驻得显式 `@Riverpod(keepAlive: true)`。判断标准：

| 数据 | 策略 | 为什么 |
|---|---|---|
| 全部分类 `categories` | **keepAlive** | 极少变、全局复用，切走切回不该重新请求 |
| 某分类商品 `categoryProducts(slug)` | **autoDispose**（默认） | slug 可能很多，离开页面就清，避免缓存无限膨胀 |
| 搜索结果 `searchResults(q)` | **autoDispose**（默认） | 关键词无穷多，更要及时清 |
| 购物车/收藏/登录态（M7/M8） | **keepAlive** | 全局唯一、活满 App 生命周期，不能被"没人看"清掉 |

- 记忆钩子：**"少而全局"用 keepAlive，"多而临时"用 autoDispose。** family 参数空间大的，默认清最稳妥。

### 3. 防抖（debounce）：属于 UI 层，不放进 provider ⭐⭐⭐

```dart
Timer? _debounce;
void _onChanged(String text) {
  _debounce?.cancel();                              // 每次输入取消上一个计时器
  _debounce = Timer(const Duration(milliseconds: 400),
      () => setState(() => _query = text));         // 停手 400ms 才提交为真正查询词
}
@override void dispose() { _debounce?.cancel(); _controller.dispose(); super.dispose(); }
```

- 防抖核心：**每敲一下就取消并重启计时器，只有"停手一段时间"才真正发请求**——
  不然每敲一个字母打一次网络，既费流量又抖动。`Timer` ≈ 可取消的 `DispatchWorkItem`。
- **职责划分**：防抖是"输入节流"，是交互层的事 → 放搜索页的 State 里。
  provider 只负责"给定关键词 → 拿结果"，保持纯粹（`searchResults(query)`）。
- 坑：`Timer` 和 `TextEditingController` 都是要 `dispose` 的可变资源，漏了会泄漏 + 页面销毁后 setState 报错。

### 4. 空关键词短路：不发无意义请求 ⭐⭐

```dart
Future<List<Product>> searchResults(Ref ref, String query) async {
  final q = query.trim();
  if (q.isEmpty) return const [];          // 空词直接空列表，不打网络
  return (await repo.searchProducts(q)).products;
}
```

- provider 和 UI 双保险：UI 侧 `hasQuery` 为空时直接显示提示、根本不 watch 结果；
  provider 侧再兜一层空词短路。省一次必然为空的请求。

### 5. 分类页的展示名走 extra、slug 走路径 ⭐

- 呼应 M5：`context.push('/category/${slug}', extra: label)`——slug 进 URL（可深链），
  中文展示名用 extra 顺手带（不进 URL）。分类页拿 `state.extra as String? ?? slug` 兜底。

## 二、新控件/API 速查表

| 概念/API | iOS 类比 | 怎么用 | 坑 |
|---|---|---|---|
| family provider | 带参数的 ViewModel 工厂 | `@riverpod ... fn(Ref ref, String slug)`，`ref.watch(p(slug))` | 每个参数独立实例+独立缓存；参数要可比较（== 决定复用哪个实例） |
| `@riverpod`（默认 autoDispose） | 无人引用即释放 | 临时/大参数空间的数据 | 想常驻要显式 keepAlive，否则页面一走就清 |
| `@Riverpod(keepAlive: true)` | 常驻单例 | 全局共享、少变的数据 | 滥用会让内存一直不释放；只给"少而全局"的用 |
| `Timer`（防抖） | 可取消的 `DispatchWorkItem` | `cancel()` 旧的 + `new` 新的，延迟触发 | 必须在 dispose 里 cancel；漏了会泄漏/setState after dispose |
| `TextField` / `TextEditingController` | `UITextField` + 手动读值 | `controller` 读写、`onChanged` 监听输入 | controller 要 dispose；`textInputAction: .search` 让回车变搜索键 |
| `ConsumerStatefulWidget` | ViewController + ViewModel | 既要 ref 又要持有可变资源（Timer/Controller）时用 | 纯展示用 ConsumerWidget 就够，别无脑上 Stateful |

## 五、练习

给搜索页加一个"最近搜索"列表：把每次真正发起的查询词（防抖后提交的那个）存进一个
本地 `List`，输入框为空时展示出来点击可回填。做完你会更清楚"防抖后提交的时机"
到底是哪一刻（不是 onChanged 的每一次）。
