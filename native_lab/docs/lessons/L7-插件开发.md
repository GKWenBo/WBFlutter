# L7 插件开发（抽出 nl_device_kit）

> 企业场景：前 6 课的桥都长在**主 app 里**——channel 在 `lib/lessons/*` 手写、原生实现塞在 `AppDelegate`/`MainActivity`、注册也得**逐条手写**。可"设备信息""埋点""网络状态"这些能力，别的 App、别的团队也要用。总不能每个项目复制粘贴一遍。本课把 **L1 的设备信息桥抽成独立插件包 `nl_device_kit`**：一个能被任意 Flutter 工程 `path`/git/pub.dev 依赖的可复用单元——企业"自家插件库"的诞生。**L1 手写版原样保留**，两页并排就是"散在 app 里的桥"vs"打包成插件的能力"的活教材。

## 一、本课要掌握什么

**1. 插件 = 把"Dart API + 原生实现"打包成一个可复用包。**

它不是新通信机制——底层还是 L1 那套 MethodChannel。变的是**工程组织**：能力从 app 里搬进 `plugins/nl_device_kit/`，有自己的 `pubspec.yaml`、`lib/`、`ios/`、`android/`，能被别的工程依赖。

**2. ★ 自动注册 vs 手写注册（本课最核心的差异）。**

| | 前 6 课（app 内嵌桥） | L7（插件） |
|---|---|---|
| 原生注册 | `AppDelegate`/`MainActivity` 里**手写** `FlutterMethodChannel(...)` + `setMethodCallHandler` | 实现 `FlutterPlugin`，由 **GeneratedPluginRegistrant 自动**调 `register(with:)`/`onAttachedToEngine` |
| 你要做的 | 每条桥手动登记一次 | **什么都不用做**——`flutter pub get` 后生成代码自动登记 |
| channel 生命周期 | 跟 App/引擎手动管 | 插件的 `onAttachedToEngine`/`onDetachedFromEngine` 托管 |

验证点：L7 页能取到设备信息，而我们**没在 AppDelegate 写一行注册**——这就是自动注册在起作用（`GeneratedPluginRegistrant.m` 里能看到 `[NlDeviceKitPlugin registerWithRegistrar:...]`）。

**3. 三层结构（官方模板默认）= 联邦插件的接缝。**

```
NlDeviceKit（公开 API，消费方只认它）
   └─委托→ NlDeviceKitPlatform.instance（抽象接口）
              └─默认是→ MethodChannelNlDeviceKit（MethodChannel 实现）
```

- 公开 API 与实现解耦：消费方调 `NlDeviceKit().getDeviceInfo()`，不关心底层。
- 抽象接口这层缝的价值：换实现（不同平台/不同后端/测试 mock）只替换 `instance`，公开 API 不变。这正是**联邦插件**的思想；完整联邦会把接口和各平台实现拆成多个包（camera/video_player 那样），本课单包演示接缝，多包留作延伸。

**4. path 依赖：企业"自家插件库"的雏形。**

app 的 `pubspec.yaml`：
```yaml
  nl_device_kit:
    path: ../plugins/nl_device_kit
```
本地开发用 `path`；对外分发改成 `git:` 或发布到 `pub.dev`（私有库用私有 pub 服务器）。

## 二、控件 / API 速查表

### 工程 & 结构（本课新东西）

| API | iOS 类比 | 用法 & 关键点 | 坑 |
|---|---|---|---|
| `flutter create --template=plugin` | 新建一个 Framework/Pod 工程 | `--platforms=ios,android --org com.wenbo -i swift -a kotlin` | 会生成 `example/`，path 依赖用不上可删 |
| `pubspec: flutter: plugin: platforms:` | 声明支持平台 + 入口类 | `ios/android` 各配 `pluginClass`（+ android `package`） | 类名/包名写错 → 自动注册找不到类 |
| `path:` 依赖 | 本地 Pod `:path =>` | app pubspec 指向 `../plugins/xxx` | 改插件后 app 要 `flutter pub get` |

### Dart 三层

| API | 说明 | 坑 |
|---|---|---|
| `NlDeviceKit`（公开门面） | 消费方唯一入口，三方法委托 platform instance | — |
| `NlDeviceKitPlatform extends PlatformInterface` | 抽象接口 + `static instance` + token 校验 | 子类要 `extends` + `MockPlatformInterfaceMixin`（测试）才过 token |
| `MethodChannelNlDeviceKit` | 默认实现，`MethodChannel('nl_device_kit')` | channel 名独立于 L1，避免撞名 |
| `export 'src/device_info.dart'` | 让消费方 import 一个库拿到 model | 忘了 export → 消费方引用不到 DeviceInfo |

### 原生侧

| API | iOS (Swift) | Android (Kotlin) |
|---|---|---|
| 入口 | `FlutterPlugin.register(with:)` | `FlutterPlugin.onAttachedToEngine(binding)` |
| 挂 handler | `registrar.addMethodCallDelegate(self, channel:)` | `channel.setMethodCallHandler(this)` |
| 拿 context | `Bundle.main` 直接用 | `binding.applicationContext`（存字段） |
| 卸载 | 框架托管 | `onDetachedFromEngine` 里 `setMethodCallHandler(null)` |

## 三、代码地图

**插件包 `plugins/nl_device_kit/`**
- `lib/nl_device_kit.dart` — `NlDeviceKit` 公开 API + `export src/device_info.dart`。
- `lib/nl_device_kit_platform_interface.dart` — `NlDeviceKitPlatform`（抽象接口 + instance + token）。
- `lib/nl_device_kit_method_channel.dart` — `MethodChannelNlDeviceKit`（channel `nl_device_kit` 三方法）。
- `lib/src/device_info.dart` — 插件的 `DeviceInfo` 模型。
- `ios/nl_device_kit/Sources/nl_device_kit/NlDeviceKitPlugin.swift` — iOS 实现（SPM 布局，搬 L1 逻辑）。
- `android/.../com/wenbo/nl_device_kit/NlDeviceKitPlugin.kt` — Android 实现。
- `test/nl_device_kit_test.dart` — 插件测试。

**App 侧**
- `pubspec.yaml` — 加 `nl_device_kit: { path: ../plugins/nl_device_kit }`。
- `lib/lessons/l7/l7_plugin_page.dart` — 用 `NlDeviceKit` 取信息，UI 对照 L1。
- `lib/lessons/lesson_registry.dart` — L7 上架。
- `test/lesson_list_test.dart` — 锁定样本顺移到 L8（懒加载列表要 scrollUntilVisible）。

## 四、测试怎么写

三层结构让测试有两个抓手：

```dart
// ① mock platform interface（这层缝的测试价值）：不碰真 channel，验证公开 API 转发
class _MockPlatform extends NlDeviceKitPlatform with MockPlatformInterfaceMixin {
  @override Future<DeviceInfo> getDeviceInfo() async => const DeviceInfo(model: 'iPhone', ...);
}
NlDeviceKitPlatform.instance = _MockPlatform();
expect((await NlDeviceKit().getDeviceInfo()).model, 'iPhone');

// ② mock channel：验证 MethodChannel 实现把原生 Map 收成强类型（对照 L1 的测法）
setMockMethodCallHandler(impl.methodChannel, (call) async => {'model': 'Pixel', ...});
expect((await impl.getDeviceInfo()).systemName, 'Android');
```

- `MockPlatformInterfaceMixin` 绕过 `PlatformInterface` 的 token 校验（否则塞 mock 会被拒）。
- App 侧：L7 页 build 冒烟（不点按钮，避免触发真 channel）。
- 插件包有自己的 `flutter test`，和 app 的分开跑。

## 五、双端对照（Swift vs Kotlin）

| 维度 | iOS (Swift) | Android (Kotlin) |
|---|---|---|
| 协议 | `NSObject, FlutterPlugin` | `FlutterPlugin, MethodCallHandler` |
| 注册入口 | `static func register(with:)` | `onAttachedToEngine(binding)` |
| 挂 handler | `registrar.addMethodCallDelegate(self, channel:)` | `channel.setMethodCallHandler(this)` |
| 分发 | `handle(_:result:)` `switch call.method` | `onMethodCall(call, result)` `when` |
| context | `Bundle.main` / `UIDevice.current` | `binding.applicationContext` |
| 自动注册 | `GeneratedPluginRegistrant` 调 `register(with:)` | 同名，v2 embedding 自动 attach |
| 清理 | 框架托管 | `onDetachedFromEngine` 解绑 |

> 本课 iOS 主讲、模拟器验证；Android 完整对照实现（本机不强制跑安卓）。

## 七、课后练习

给插件加一个新能力 `getDeviceModelName()`（返回更细的机型串，如 iOS 的 `iPhone16,2`），走一遍完整回路：

1. `platform_interface` 加抽象方法 `Future<String> getDeviceModelName()`；
2. `method_channel` 实现里加 `invokeMethod('getDeviceModelName')`；
3. iOS 用 `utsname`/`sysctlbyname("hw.machine")`、Android 用 `Build.MODEL`/`Build.DEVICE` 各加一个 case；
4. `NlDeviceKit` 暴露该方法；
5. app L7 页加一行显示；
6. 补一条 mock platform 的测试。

练的是"插件里加方法 → 接口/实现/两端/调用点一起动"的完整约束——这正是插件 API 演进的日常。
