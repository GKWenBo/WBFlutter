# L2 数据编解码与复杂参数

> 企业场景：埋点桥——Flutter 的埋点统一走原生已接好的统计 SDK（友盟/GA/自研）。事件名 + 一坨属性从 Dart 灌到原生，原生回执，还能把缓存的事件批量拉回来。这一课把 MethodChannel 底下的"搬运工" **StandardMessageCodec** 彻底讲透。

## 一、本课要掌握什么

**1. StandardMessageCodec 是谁、干什么。**

L1 你只传了方法名。这课要传**结构化数据**：`invokeMethod('logEvent', {嵌套 Map})`。这个 Map 怎么变成二进制、跨过线程、再在 Swift 侧变回 `NSDictionary`？靠的就是 MethodChannel 默认挂的编解码器 **StandardMessageCodec**。

心智模型：**codec = 一张固定的类型对照表 + 一套二进制打包规则**。它只认表上的类型（null/bool/int/double/String/List/Map/几种 TypedData）。你塞一个它不认识的东西（比如自定义类实例、`DateTime`、`Uint16List`）进去，**编码那一刻就抛**，根本过不了桥。所以 L1 我们才强调"过桥前先 `toMap()` 拍平"——自定义类必须先降解成 codec 认识的裸结构。

iOS 类比：像 `NSKeyedArchiver`/`Codable` 之于对象持久化——只有能被序列化的类型才能过关，中间是二进制而不是 JSON 文本（**比 JSON 快、且能零拷贝传字节**，见下文二进制）。

**2. ★ 类型映射表（本课核心，必须背下来）。**

| Dart | Swift (iOS) | Kotlin (Android) | 说明 |
|---|---|---|---|
| `null` | `NSNull` / `nil` | `null` | |
| `bool` | `NSNumber` → `Bool` | `Boolean` | Swift 侧数字都裹在 NSNumber 里 |
| `int`（≤32 位） | `NSNumber` → `Int` | `Integer` (`Int`) | |
| `int`（>32 位） | `NSNumber` → `Int` | **`Long`** | ⚠️ Android 大整数是 Long，见坑 2 |
| `double` | `NSNumber` → `Double` | `Double` | ⚠️ 与 int 不可混，见坑 1 |
| `String` | `String` (`NSString`) | `String` | |
| `Uint8List` | `FlutterStandardTypedData` | `ByteArray` | 二进制零拷贝 |
| `Int32List`/`Int64List`/`Float64List` | `FlutterStandardTypedData` | `IntArray`/`LongArray`/`DoubleArray` | 数值数组高效传 |
| `List<Object?>` | `NSArray` / `[Any]` | `List<Any?>` | 元素递归编码 |
| `Map<Object?,Object?>` | `NSDictionary` / `[AnyHashable: Any]` | `Map<Any?,Any?>` | 键值都是 `Object?` |

**记忆钩子**：Swift 侧数字**全是 `NSNumber`**（取值 `.intValue`/`.doubleValue`/`.boolValue`），所以 iOS 几乎没有数字类型坑；Kotlin 侧数字**按大小/精度分裂成 `Integer`/`Long`/`Double`**，坑都在这。回 Dart 时，容器里的键值静态类型都是 `Object?`，**必须 `.cast` 或 `invokeMapMethod`/`invokeListMethod` 收窄**才好用。

**3. 三大类型事故（本课暗线，延续 L1 的失败路径教学）。**

- **坑 1 · int/double 不可混。** codec **不做隐式数字转换**。原生约定回 `int` 却回了 `double`（比如把 `3` 写成 `3.0`，或返回 `NSTimeInterval`），Dart 侧 `invokeMethod<int>` 在 `as int` 时抛 **`TypeError`**——崩在 **Dart 端**。这就是 L1 `getSystemUptime` 为什么必须用 `double` 接（`ProcessInfo.systemUptime` 是 `Double`）。本课测试 `类型坑：原生返回 double 却用 <int> 接` 亲手复现了它。
- **坑 2 · Android 的 Int/Long 边界。** Dart `int` 是 64 位；codec 编码时按值大小选 int32 或 int64。到 Kotlin 侧，小值是 `Integer`、大值（超 int32）是 `Long`。所以 `map["ts"] as Int` 对一个毫秒时间戳（大数）会 **`ClassCastException`**，稳妥写法 `(map["ts"] as Number).toLong()`。iOS 侧 `NSNumber` 统一，无此坑。
- **坑 3 · `Map<Object?,Object?>` 直接当 `Map<String,dynamic>` 用会崩。** 原生传回的容器键值都是 `Object?`，`as Map<String, Object?>` 是失败的强转。要用 `.cast<String, Object?>()`（运行时逐个检查）或顶层用 `invokeMapMethod`/`invokeListMethod`。本课 `AnalyticsEvent.fromMap` 和 `fetchLoggedEvents` 都在做这个收窄。

**4. 二进制为什么用 `Uint8List` 而不是 base64 String。**

传字节流（缩略图、protobuf、加密包）时，`Uint8List` 走 codec 的 **typed-data 分支**：直接按字节打包，**零拷贝、无膨胀**。若退化成 base64 `String`：体积涨 ~33%、两端各多一次编解码 CPU。所以规矩是——**二进制就用 `Uint8List`/`ByteArray`，别 base64**。

## 二、控件 / API 速查表

### Dart 侧

| API | iOS 类比 | 用法 & 关键点 | 坑 |
|---|---|---|---|
| `invokeMethod(method, arguments)` | 带参 RPC | 第二参 = 传给原生的数据，任意 codec 支持的嵌套结构 | 塞不支持类型→编码期抛 |
| `invokeMapMethod<K,V>()` | 返回收窄成 Map | 顶层是 Map 时用，省手动 cast | 仍可能返回 null |
| `invokeListMethod<T>()` | 返回收窄成 List | 顶层是 List 时用 | 元素仍是 `Object?`，要再 cast |
| `.cast<K,V>()` | 类型收窄 | 把 `Map<Object?,Object?>` 逐项收成 `Map<String,Object?>` | 与 `as` 不同：cast 是运行时逐个检查 |
| `Uint8List` | `NSData` | 二进制载荷 | 只有它零拷贝，别用 base64 |

### Swift 侧

| API | 用法 & 关键点 |
|---|---|
| `call.arguments as? [String: Any]` | Dart Map → NSDictionary；数字取 `.intValue`/`.doubleValue` |
| `call.arguments as? FlutterStandardTypedData` | Dart `Uint8List` 的原生落点，`.data` 是 `Data` |
| `result(buffer)` | `[[String:Any]]` 自动编码回 Dart 的 `List<Map>` |

### Kotlin 侧

| API | 用法 & 关键点 |
|---|---|
| `call.arguments as? Map<String, Any?>` | 数字用 `(x as Number).toLong()/.toDouble()`，别裸 `as Int` |
| `call.arguments as? ByteArray` | Dart `Uint8List` 的原生落点，直接 `.size` |

## 六、课后练习

给埋点桥加**批量上报** `logBatch(List<AnalyticsEvent>)`（一次把多条事件灌下去，减少过桥次数——真实 SDK 常这么做）：
- Dart：`AnalyticsBridge.logBatch(List<AnalyticsEvent> events) -> Future<int>`，`invokeMethod('logBatch', events.map((e) => e.toMap()).toList())`，返回 buffer 新总数；
- Swift：`case "logBatch": guard let list = call.arguments as? [[String: Any]] ...; buffer.append(contentsOf: list); result(buffer.count)`；
- Kotlin：`"logBatch" -> { val list = call.arguments as? List<Map<String, Any?>>; ... analyticsBuffer.addAll(list); result.success(analyticsBuffer.size) }`；
- 补一条 mock 单测：传 3 条事件，mock 原生返回 3，断言 `logBatch()` 得到 3，并校验原生收到的 `call.arguments` 是长度为 3 的 List。

（提示：这题练的是"List 作为顶层参数"——注意三端把 `List<Map>` 转成对应容器类型时的写法差异。）

> ✅ **本练习已在仓库落地为参考实现**（本轮由讲师完成）：Dart [analytics_bridge.dart](../../app/lib/lessons/l2/analytics_bridge.dart) `logBatch`、Swift/Kotlin 各 `case "logBatch"`、单测 `课后练习·批量上报`。三端要点对照：
> - **顶层是 List 而非 Map**：`invokeMethod('logBatch', events.map((e) => e.toMap()).toList())`；codec 顶层直接支持 List。
> - **Swift**：`call.arguments as? [[String: Any]]`（NSArray→Swift 数组），`buffer.append(contentsOf:)`。
> - **Kotlin**：`call.arguments as? List<Map<String, Any?>>`（需 `@Suppress("UNCHECKED_CAST")`），`analyticsBuffer.addAll(list)`。
> - 三端都回 `buffer.count/size`（累加后的新总数），Dart 侧 `invokeMethod<int>` 接。
