# L4 页面级混合 + 权限

> 企业场景：商品页点"扫码"，唤起原生扫码页（真机上是相机预览 + 二维码识别），扫到码回传给 Flutter 去查商品。前三课都是"数据/流过桥"，这一课第一次让原生**接管整块屏幕**，还第一次碰**运行时权限**。

## 一、本课要掌握什么

**1. 页面级混合 = 原生页面接管整屏，不是数据过桥。**

L1（读设备信息）、L2（埋点）、L3（网络流）本质都是"数据在两端来回传"，Flutter 的 UI 始终自己占着屏幕。**本课不一样**：Flutter 让原生 `present` 一个 `UIViewController` **盖住整屏**，用户在原生页里操作，操作完 `dismiss`、把结果回传。屏幕的主权临时交给了原生。

这跟你 iOS 的老本行几乎一样——`present(vc, animated:)` 弹一个页、完事 `dismiss` 带个结果回来——只是这次**触发点在 Flutter**。

> 埋个预告：L6 的 `PlatformView` 是另一条路——**视图级混合**，把原生 view（如 `MKMapView`）嵌进 Flutter 的 widget 树里，和 Flutter 控件同屏共存。**页面级（本课）= 原生页盖住整屏；视图级（L6）= 原生 view 是 Flutter 树里的一个小块。** 到 L6 会正式对比选型。

**2. ★ 延迟的 `FlutterResult`（本课最核心的新招）。**

前三课原生 handler 里**立刻**就 `result(...)` 回值。本课不行——`present` 出去后，得等用户扫完/取消才有结果。所以：

- 把那次 `result` **暂存成 `pendingResult`**，handler 先 return；
- 用户在原生页点了按钮 → `dismiss` 的 completion 里，才调 `pendingResult?(code)` 把值回给 Flutter；
- **`FlutterResult` 只能调用一次**——所以要 ① 单次在飞行守卫（上一次没回就拒新请求，回 `ALREADY_SCANNING`）② 回完把 `pendingResult` 置空。

从 Flutter 看：`await ScanBridge.scan()` 这行**一直挂起**，present 出去期间 Flutter 页仍在栈里（只是被盖住），直到原生页 dismiss、结果回来，`Future` 才完成。

```
Dart:  await scan()  ────────────────────────────┐（一直挂起）
  → 原生 handler：暂存 result 为 pendingResult
  → present 原生扫码页（盖住整屏）
     用户点"扫到 SKU-10086"
  → dismiss(completion:)
       completion 里 pendingResult?("SKU-10086"); pendingResult = nil
Dart:  await 返回 ScanSuccess("SKU-10086") ◄───────┘（Future 完成）
```

**3. 运行时权限三态 + 拒绝降级 + Info.plist。**

扫码要相机权限。iOS 相机权限有三种状态，分别处理：

| 状态 | 含义 | 处理 |
|---|---|---|
| `.authorized` | 已授权 | 直接 present 扫码页 |
| `.notDetermined` | 还没问过 | `requestAccess` 弹系统授权框，回调里按 granted 决定 present 还是回 `PERMISSION_DENIED` |
| `.denied` / `.restricted` | 拒过 / 被限制 | 直接回 `PERMISSION_DENIED`，Flutter 侧提示"去设置开启" |

两个坑：① `requestAccess` 回调**在任意线程**，`present` 必须切回主线程（同 L1/L3 的线程纪律）；② **`Info.plist` 必须配 `NSCameraUsageDescription`**，否则一调 `requestAccess` **直接崩**（这是苹果强制的隐私声明）。

**4. 测试"倒退"回 `setMockMethodCallHandler`。**

L3 刚学流用 `setMockStreamHandler`。本课**回到** L1/L2 的 `setMockMethodCallHandler`——因为页面级混合本质仍是"**一次请求、一个结果**"（MethodChannel），只是结果来得晚。选型口诀延续：**"present 出去等一个结果回来" → 还是一问一答；"盯着会变的东西持续收" → 才是流。**

## 二、控件 / API 速查表

### Dart 侧

| API | iOS 类比 | 用法 & 关键点 | 坑 |
|---|---|---|---|
| `invokeMethod<String>('scan')` | 带 completion 的异步调用 | 返回的 `Future` 会挂起到原生页 dismiss 才完成 | 结果"延迟"，不是立刻回 |
| `sealed class ScanOutcome` | Swift `enum` with associated values | 把"回串/回null/抛错"三种情形收成 `ScanSuccess`/`ScanCancelled`/`ScanPermissionDenied` | UI `switch` 漏分支编译期报错 |
| `on PlatformException catch` | `catch let error as NSError` | 桥接层把 `PERMISSION_DENIED` 转成 `ScanPermissionDenied`，其它错误 `rethrow` | 别一把吞掉所有错误 |

### Swift 侧

| API | iOS 类比 | 用法 & 关键点 |
|---|---|---|
| 暂存 `FlutterResult` | completion handler 存成属性 | present 前 `pendingResult = result`，dismiss 回调里才调；**只能调一次** |
| `AVCaptureDevice.authorizationStatus(for: .video)` | —（同名） | 读当前相机权限三态 |
| `AVCaptureDevice.requestAccess(for: .video){ }` | —（同名） | 首次弹授权框；回调在任意线程，present 要 `DispatchQueue.main.async` |
| `present(_:animated:)` / `dismiss(animated:)` | —（同名） | 页面级混合的核心；`dismiss` 的 completion 里再回传结果 |
| `Info.plist` `NSCameraUsageDescription` | —（同名） | **不配直接崩**；字符串是给用户看的用途说明 |
| 取 topViewController | `keyWindow?.rootViewController` | Flutter+Scene 世界要从前台 `UIWindowScene.keyWindow`（iOS 15+）找最顶层 VC 再 present |

## 四、双端对照

| 对照点 | iOS (Swift) | Android (Kotlin) |
|---|---|---|
| 打开原生页 | `present(UIViewController)` | `startActivity` / `ActivityResultLauncher.launch(Intent)` |
| 结果回传 | 暂存 `FlutterResult`，`dismiss` 回调里回 | 暂存 `MethodChannel.Result`，`onActivityResult`/launcher 回调里回 |
| 为什么都要暂存 | 结果延迟到用户操作后才有——两端同理 | 同左 |
| 运行时权限 | `AVCaptureDevice.requestAccess` + `NSCameraUsageDescription` | `ActivityResultContracts.RequestPermission` + Manifest `<uses-permission CAMERA>` |
| 一个易错点 | present 要找对 topViewController（Scene 世界） | ① launcher 必须在 `STARTED` 之前注册（放字段初始化）；② `MainActivity` 得继承 `FlutterFragmentActivity`——普通 `FlutterActivity`(=`android.app.Activity`) 没有 `registerForActivityResult`；③ 新 Activity 要在 Manifest 登记 |

## 六、课后练习

给 `scan()` 加一个入参 `hint`（如"请对准商品条码"），传给原生扫码页当标题——练"**打开原生页时携带入参**"：

- Dart：`scan({String? hint})` → `invokeMethod('scan', {'hint': hint})`；
- Swift：`call.arguments` 取 `hint`，设成 `MockScannerViewController` 的 `title`；
- 补一条 mock 测试：`mockScan` 里断言 `call.arguments['hint']` 收到了透传的值。

## 附：构建注意（本课真实踩到，重要）

本仓库在 **Xcode 27 Beta** 下构建 L4 时踩到两点，记下来省得你以后困惑：

1. **iOS 部署目标要 ≥ 15.0。** 工程原本 `IPHONEOS_DEPLOYMENT_TARGET = 13.0`，Xcode 27 的模拟器 SDK 最低要 15.0，全新构建会报错。已把 `project.pbxproj` 三处改成 `15.0`——正好本课 `topViewController()` 用的 `UIWindowScene.keyWindow` 就是 iOS 15+ API。
2. **`flutter build ios` 外壳偶发误报。** 这套 Xcode Beta 下 `flutter build`/`flutter run` 的准备脚本会误报 `Flutter.framework 缺 arm64/x86_64`（`lipo` 其实都有，只是顺序反了）。绕法：直接 `xcodebuild -workspace Runner.xcworkspace -scheme Runner -sdk iphonesimulator ...` 构建正常，产物 `Runner.app` 用 `xcrun simctl install` 装机即可。Dart 代码零编译错误，纯属工具链外壳 bug。
