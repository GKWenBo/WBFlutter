# L3 自测答案

> 配套 [L3-EventChannel网络状态推流.md](../L3-EventChannel网络状态推流.md)。答案对应仓库当前真实代码，链接可点开核对。

## Q1. EventChannel 和 MethodChannel 分别什么时候用？给两个各自的典型场景，说出判断口诀。

- **MethodChannel = 请求-响应，一问一答。** Dart `invokeMethod` 发一次、原生 `result` 答一次，`Future` 完成即结束。典型场景：读设备信息（L1 `getDeviceInfo`）、埋点上报（L2 `logEvent`）——**做一次、拿个结果**的活。
- **EventChannel = 订阅-推送，持续收。** Dart 订阅一次，原生**想推就推**，是一条单向（原生 → Dart）持续的流。典型场景：网络状态变化（本课）、电量/传感器/下载进度——**盯着一个会变的东西**。
- **判断口诀：「做一次拿个结果」→ MethodChannel；「盯着一个会变的东西」→ EventChannel。** 反例警示：想"网络变了通知我"却用 MethodChannel，就只能每 100ms 轮询一次——费电又有延迟，正是 EventChannel 存在的理由。

## Q2. `onListen` / `onCancel` 分别在 Dart 侧什么动作时被触发？各对应 iOS 的什么 API？忘了 `onCancel` 会有什么后果？

| 钩子 | Dart 侧触发动作 | iOS 类比 | 该做什么 |
|---|---|---|---|
| `onListen(args, events)` | `receiveBroadcastStream().listen(...)`——本课即 `StreamBuilder` **挂载** | `addObserver` / Combine `.sink` | 拿住 `events`(EventSink)，启动监听源（新建 `NWPathMonitor` 并 `start`） |
| `onCancel(args)` | 取消订阅——本课即 `StreamBuilder` **dispose**（页面退出） | `removeObserver` / `AnyCancellable.cancel()` | 拆掉监听源（`monitor.cancel()`），防泄漏 |

- **忘了 `onCancel` 的后果 = 泄漏。** 页面都关了，`NWPathMonitor`（Android 的 `NetworkCallback`）还在后台跑、还在往一个没人要的 sink 推事件——白耗电、白占资源，跟 iOS 忘了 `removeObserver` 一个性质。见 [AppDelegate.swift](../../../app/ios/Runner/AppDelegate.swift) 的 `NetworkStatusStreamHandler.onCancel`。

## Q3. 为什么 `NWPathMonitor` 回调里往 sink 推事件前要 `DispatchQueue.main.async`？不切线程会怎样？

- `NWPathMonitor` 的 `pathUpdateHandler` 是在你 `start(queue:)` 传进去的**那条后台队列**上回调的（本课是 `com.wenbo.native_lab.netmonitor`）。而 Flutter 的 platform channel 约定**事件要在平台主线程投递**。
- 不切线程直接 `events(...)`：轻则行为未定义、事件时序错乱，重则触发 Flutter 引擎的线程断言而**崩溃**。所以必须 `DispatchQueue.main.async { events(...) }` 回主线程再投。
- 这跟 L1「handler 跑在主线程、耗时活要挪到后台」是**同一条线程纪律的两面**：耗时任务往后台挪，但**回调结果/事件必须回主线程**。Android 侧对应 `runOnUiThread { events.success(...) }`，一模一样的道理。见 [AppDelegate.swift](../../../app/ios/Runner/AppDelegate.swift) `pathUpdateHandler` 与 [MainActivity.kt](../../../app/android/app/src/main/kotlin/com/wenbo/native_lab/MainActivity.kt) 的 `onAvailable`/`onLost`。

## Q4. 本课页面用 `StreamBuilder` 而不是 `StatefulWidget` 手动管订阅，`StreamBuilder` 替你托管了订阅的哪两件事？

`StreamBuilder` 把订阅的**生命周期**跟 widget 的生命周期绑在一起，替你做了两件事：

1. **挂载时自动 `listen`**——widget 第一次 build 就订阅了传入的 `stream`（→ 触发原生 `onListen` 开始监听），每来一个事件自动 `setState` 重建 `builder`，你不用手写 `setState`。
2. **dispose 时自动 `cancel`**——widget 销毁时取消订阅（→ 触发原生 `onCancel` 拆监听），你不用在 `dispose()` 里手动 `subscription.cancel()`，也就不会忘、不会泄漏。

所以本课页面用 `StatelessWidget` 就够了（见 [l3_network_page.dart](../../../app/lib/lessons/l3/l3_network_page.dart)）——省掉了 `initState` 订阅 + `dispose` 取消那套样板。代价是每次 `snapshot` 要自己判 `hasError` / `hasData`（首帧 `hasData` 为 false，要处理 loading）。

## 附一：本课真实踩过的 bug（第二次进页面一直 loading）

**现象：** 第一次进 L3 正常，返回再进就卡在转圈。
**根因：** `NWPathMonitor` 一旦 `cancel()` 就报废、**不能 `start()` 重启**。原来 handler 用一个 `let monitor` 反复用——第一次 `onCancel` 把它 cancel 了，第二次 `onListen` 复用这个死 monitor，`pathUpdateHandler` 再也不回调 → sink 永不吐值 → `StreamBuilder` 卡在 `!hasData`。
**修法：** 改成 `var monitor: NWPathMonitor?`，**每次 `onListen` 新建**一个，`onCancel` 里 `cancel()` 后置 `nil`（见 [AppDelegate.swift](../../../app/ios/Runner/AppDelegate.swift) 的 `NetworkStatusStreamHandler`）。
**引申：** 凡是"一次性、cancel 后不可复用"的监听源，都要在 `onListen` 里建、`onCancel` 里销毁，别在 handler 初始化时建一个反复用。Android 侧本课没这毛病——因为每次 `onListen` 都 `new` 了新的 `NetworkCallback`，正好是正确姿势的对照。

## 附二：课后练习核对（信号强度：String → Map 升级）

要求把原生推的从裸字符串升级成 `Map{'type','level'}`，并加 `NetworkInfo`。核对点：

| 端 | 实现 | 位置 |
|---|---|---|
| Dart 模型 | `NetworkInfo { NetworkStatus status; int level }` + `fromMap`（`level` 用 `num?.toInt()` 兜 int/double 坑） | [network_status.dart](../../../app/lib/lessons/l3/network_status.dart) |
| Dart 桥 | `statusStream()` 改回 `Stream<NetworkInfo>`，`map` 里 `NetworkInfo.fromMap(raw as Map<Object?,Object?>?)` | [network_bridge.dart](../../../app/lib/lessons/l3/network_bridge.dart) |
| Dart 页面 | `StreamBuilder<NetworkInfo>`，新增 `_SignalBars` 画信号格、副标题显示 `信号强度：n / 3` | [l3_network_page.dart](../../../app/lib/lessons/l3/l3_network_page.dart) |
| Swift | `events(["type": type, "level": level])`（`NWPath` 拿不到真实强度，占位满格 3） | [AppDelegate.swift](../../../app/ios/Runner/AppDelegate.swift) |
| Kotlin | `events.success(mapOf("type" to "wifi", "level" to 3))` | [MainActivity.kt](../../../app/android/app/src/main/kotlin/com/wenbo/native_lab/MainActivity.kt) |
| 测试 | 新增「单个 Map 事件 → NetworkInfo（status==wifi && level==3）」+ Map 序列 + 兜底 | [l3_network_test.dart](../../../app/test/l3_network_test.dart) |

**要点：** 流里的每个事件也可以是 Map/List——codec 规则和 L2 的方法参数**完全一样**。这题把 L2 的"复杂结构编解码"和 L3 的"流"合了起来。
