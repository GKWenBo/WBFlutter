# L7 自测答案

**1. 插件 channel 为什么要独立于 L1？同名会怎样？**

L1 的桥在 `AppDelegate`/`MainActivity` 里**手写注册**在 `com.wenbo.native_lab/device_info` 上；插件的原生代码由 GeneratedPluginRegistrant **自动注册**。如果插件也用这个名，就有**两处**给同一个 channel 名挂 handler——后挂的覆盖先挂的，行为取决于注册顺序，极易出诡异 bug。用独立名 `nl_device_kit` 让两套桥井水不犯河水，也正好让"L1 手写 vs L7 插件"能并排共存对照。

**2. 自动注册的时机与完成者？少写了什么？**

Flutter 引擎启动、执行 `GeneratedPluginRegistrant.register(with:)`（iOS 在 AppDelegate 里那一行；Android v2 embedding 自动）时，会遍历所有依赖的插件，逐个调用它们的 `register(with:)`（iOS）/ `onAttachedToEngine`（Android）。`GeneratedPluginRegistrant` 是 `flutter pub get` 根据依赖的插件**自动生成**的。相比前 6 课，我们**少写了每条桥的手动登记代码**——插件把"登记"这件事标准化、自动化了。

**3. 三层各负责什么？去掉中间层会丢什么？**

- `NlDeviceKit`（公开门面）：消费方唯一入口，稳定的对外 API，屏蔽底层实现。
- `NlDeviceKitPlatform`（抽象接口）：定义"这个插件能做什么"的契约 + 持 `instance`。
- `MethodChannelNlDeviceKit`（实现）：具体怎么做——走 MethodChannel。

去掉中间抽象层（让门面直接调 MethodChannel）也能跑，但会**丢掉可替换性**：无法在不改公开 API 的前提下换实现（换平台实现、换后端、或测试里塞 mock）。这层缝就是"联邦插件"的接缝，也是测试能 mock 的支点。

**4. 为什么要 `MockPlatformInterfaceMixin`？绕过了什么？**

`PlatformInterface` 构造时用一个私有 `_token` 做身份校验，`set instance` 会 `verifyToken`——防止外部乱塞一个"冒充"的实现（企业里防止第三方插件破坏契约）。但测试里我们**故意**要塞 mock，`MockPlatformInterfaceMixin` 就是官方给的"合法后门"，绕过 token 校验，让 mock 能被 `instance =` 接受。

**5. path vs pub.dev/git 依赖？**

- `path`：本地开发阶段——插件和 app 在同一 monorepo，改插件即时生效，无需发布。
- `git`：插件放在某个 git 仓库，多个项目共享但还没/不想上公共仓库（企业私有仓库常用）。
- `pub.dev`：正式发布、对外公开分发（私有场景用私有 pub 服务器）。
本课停在 `path`，因为目标是"抽出可复用单元"这件事本身，不涉及分发。

**6. 联邦插件是什么？本课是不是？何时才拆多包？**

**联邦插件** = 把"抽象接口""app 面向门面""各平台的具体实现"拆成**多个独立包**，这样某个平台的实现可以由**第三方独立提供/替换**（如 `path_provider` + `path_provider_android` + `path_provider_ios` + `path_provider_platform_interface`）。本课是**单包三层**——有联邦的**接缝**（platform interface 这层）但没拆包，所以**不是**完整联邦。只有当你需要"让别人能独立发布某平台实现""平台数量多且各自演进"时，才值得拆成完整多包；小插件单包足矣（YAGNI）。
