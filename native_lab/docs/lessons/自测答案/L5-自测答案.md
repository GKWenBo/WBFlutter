# L5 自测答案 — Pigeon 类型安全生成

> 配套 [L5-Pigeon类型安全生成.md](../L5-Pigeon类型安全生成.md) 第六节自测清单。

## 1. Pigeon 相比手写 MethodChannel 省了什么？编译期能抓住哪些手写抓不住的错？

**省了三样手写活：**
- **channel 名 / 方法名字符串**：手写要两端逐字符对齐 `'com.wenbo.native_lab/device_info'` + `'getDeviceInfo'`，Pigeon 里 channel 名由生成代码管理，你调的是 `host.getDeviceInfo()` 真方法。
- **参数/返回的 Map 拆装**：手写 `map['model'] as String`、还要背 L2 那套 codec 类型映射坑（int/double、Integer/Long、Map cast）；Pigeon 给你强类型类 `DeviceInfoData`。
- **两端同步**：手写改一处要手动同步另外两端；Pigeon 改契约重跑，三端一起变。

**编译期能抓住（手写只能运行期崩）：**
- 方法名打错 / 改名 → 手写运行期 `MissingPluginException`/`notImplemented`；Pigeon **编译报错**。
- 字段名打错、少传字段、类型不匹配 → 手写运行期 `TypeError`/`as` 崩；Pigeon 因为是强类型类，**编译报错**。
- 原生侧漏实现某个方法 → 手写 `notImplemented` 运行期才知道；Pigeon 不实现协议/接口方法**编译报错**。

一句话：**Pigeon 把"运行期才炸的字符串/类型错误"提前到了编译期。**

## 2. @HostApi 和 @FlutterApi 分别对应哪一课的方向？各自生成什么？

- `@HostApi()` = **Flutter → 原生**，对应 **L1**（Dart 调原生取设备信息）。
  - Dart 侧生成：可直接调的 `DeviceInfoHostApi()` 类（内部封了 channel + codec）。
  - 原生侧生成：要你实现的 `protocol`(iOS) / `interface`(Android) + 一个 `Setup.setUp(...)` 挂载函数。
- `@FlutterApi()` = **原生 → Flutter**（反向），对应 **L3** 的方向（原生主动推给 Flutter）。
  - 原生侧生成：可直接 `new` 的 `DeviceEventFlutterApi` 调用端。
  - Dart 侧生成：一个 abstract `DeviceEventFlutterApi` + `setUp(接收者)`，你实现 `onBatteryChanged` 当接收端。

## 3. pigeon 27 里怎么给桥写测试？替代了 L1 里手写的什么？

pigeon 27 **弃用了生成 mock host**（`dartHostTestHandler`），官方改推 **"fake 生成的 Dart API"**：
- 继承生成的 `DeviceInfoHostApi`、覆写方法当假原生，通过 **构造函数注入** 进 bridge（`DeviceInfoPigeonBridge(hostApi: _FakeHost())`）。
- 这替代了 L1 里手写的 `setMockMethodCallHandler` + 手拼返回 `Map`。好处：桩是强类型的（覆写方法返回 `DeviceInfoData`，字段打错编译报错），比"手拼一个 Map 假装是原生回值"更贴近真实、更安全。
- 反向则直接调 `bridge.onBatteryChanged(BatteryInfo(...))` 验证 `batteryStream`。
- 页面测试（页面自己 new bridge，注入不进去）用 `setMockDecodedMessageHandler` 在 Pigeon 私有 `BasicMessageChannel` 上顶替，回复格式是 `[返回值]` 的 List。

## 4. Pigeon 反向（FlutterApi）和 L3 EventChannel 都能"原生推 Flutter"，差别与取舍？

| | L3 EventChannel | L5 FlutterApi |
|---|---|---|
| 形态 | 一条 Stream，有 `onListen`/`onCancel` 生命周期 | 普通方法回调，原生主动调 |
| 事件类型 | 裸值 / `Map`，Dart 侧手拆 | 强类型 `BatteryInfo` |
| 订阅管理 | `StreamBuilder` 自动 listen(挂载)/cancel(dispose) | 自己调 `start/stopBatteryUpdates` 控制何时开始/停止 |

**取舍**：天生就是"持续订阅一个数据源"（网络、传感器）→ EventChannel 更顺，`StreamBuilder` 一挂全自动。要**类型安全**、或反向调用不止一种事件（多个方法）→ FlutterApi 更合适，但订阅的开/关要自己用方法管。本课两者都体验了一遍。

## 5. 改了契约字段名不重跑生成会怎样？为什么说这正是 Pigeon 的价值？

- 只改契约文件、不重跑：生成物没变，代码照旧编过、按旧字段跑——**契约和实际生成脱节了**，等于没改。所以规矩是**改契约必重跑 `dart run pigeon`**。
- 重跑之后才见价值：假设把 `model` 改成 `deviceModel`，生成物里字段名跟着变，于是**所有还在用旧 `.model` 的调用点全部编译报错**，编译器逐个点名让你改。手写 `Map` 做不到这点——`map['model']` 改没改全靠人肉记，漏一个就运行期静默拿到 null/崩。**"改一处、编译器push 着你改全三端"正是 Pigeon 对团队协作的核心价值。**

## 课后练习核对（getBatteryInfo 一次性取电量）

- **契约**：`DeviceInfoHostApi` 里加 `BatteryInfo getBatteryInfo();`（不加 `@async` 就是同步返回）。
- **重跑生成**后，iOS 协议多一个 `getBatteryInfo() throws -> BatteryInfo`、Android interface 多一个 `fun getBatteryInfo(): BatteryInfo`——**不实现直接编译报错**，这就是"契约驱动"的手感。
- **实现**：复用电量读取逻辑，返回 `BatteryInfo`（iOS `UIDevice`、Android `BatteryManager`）。
- **测试**：`_FakeHost` 覆写 `getBatteryInfo` 返回固定值，断言 `bridge.getBatteryInfo()` 正确。
- 完成后可把这个方法接到页面"读一次电量"按钮，和订阅式的电量流做对比：**一次性拉 vs 持续推**——正好复习 L1(拉) 与 L3(推) 的选型口诀。
