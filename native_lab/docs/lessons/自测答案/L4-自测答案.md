## Q1. 页面级混合和 L1–L3 的"数据过桥"差在哪？和 L6 的 PlatformView（视图级）又差在哪？各举一句话场景。

- **vs L1–L3 数据过桥：** L1–L3 是"数据/流在两端来回传"，Flutter 的 UI 全程自己占着屏幕（读设备信息、埋点、网络流都只是拿数据）。**页面级混合是原生 `present` 一个 `UIViewController` 盖住整屏**，屏幕主权临时交给原生，用户在原生页里操作完再回传结果。一句话场景：**扫码页/原生支付页/系统相册选择器**——需要一整个原生页面接管交互。
- **vs L6 PlatformView（视图级）：** PlatformView 是把原生 view（如 `MKMapView`）**嵌进 Flutter 的 widget 树**里，和 Flutter 控件同屏共存、一起滚动。**页面级 = 原生页盖住整屏；视图级 = 原生 view 只是 Flutter 树里的一小块。** 一句话场景：**页面里嵌一小块原生地图/原生视频播放器**，周围还是 Flutter 控件。
- 选型口诀：**要"一整页原生交互、拿个结果回来" → 页面级（本课）；要"原生 view 和 Flutter 控件混排同屏" → 视图级（L6）。**

## Q2. 为什么本课的 `FlutterResult` 要暂存、不能在 handler 里立刻回？"只能调用一次"这条没守住会怎样？

- **为什么暂存：** L1/L2 的 handler 里数据当场就有，`result(...)` 立刻回。但扫码的结果要**等用户在原生页扫完/取消才有**——handler 里 `present` 出去后就 return 了，此刻根本没有结果可回。所以把 `result` 存成 `pendingResult`，等原生页 `dismiss` 的回调里才调 `pendingResult?(code)`。见 [AppDelegate.swift](../../../app/ios/Runner/AppDelegate.swift) 的 `presentScanner` / `finish(with:)`。
- **`FlutterResult` 只能调一次**，没守住的两类具体后果：
  1. **重复调用**：present 期间用户又触发一次 scan，若不拦，第二次的 `result` 会覆盖 `pendingResult`，第一次的调用**永远不回**（Flutter 那个 `Future` 永久挂起）。所以加了**单次在飞行守卫**：`pendingResult != nil` 就直接回 `ALREADY_SCANNING`。
  2. **回两次**：同一个 `result` 调用两次（比如 dismiss 回调和错误分支都回），Flutter 引擎行为未定义/报错。所以 `finish(with:)` 回完立刻 `pendingResult = nil`。
- Android 侧同理：`pendingResult: MethodChannel.Result?` + `finishScan` 回完置空，见 [MainActivity.kt](../../../app/android/app/src/main/kotlin/com/wenbo/native_lab/MainActivity.kt)。

## Q3. 相机权限三态分别怎么处理？`NSCameraUsageDescription` 不配会怎样？`requestAccess` 回调里 present 前为什么要切主线程？

三态处理（见 [AppDelegate.swift](../../../app/ios/Runner/AppDelegate.swift) `handleScan`）：

| 状态 | 处理 |
|---|---|
| `.authorized` | 已授权，直接 `presentScanner` |
| `.notDetermined` | 还没问过 → `requestAccess` 弹系统授权框，回调里 `granted ? present : 回 PERMISSION_DENIED` |
| `.denied` / `.restricted` | 拒过 / 被限制 → 直接回 `PERMISSION_DENIED`，Flutter 提示"去设置开启" |

- **`NSCameraUsageDescription` 不配会怎样：** 一调 `AVCaptureDevice.requestAccess(for: .video)` **直接崩**（Apple 强制：访问隐私敏感资源前必须在 Info.plist 声明用途字符串，否则运行时终止）。见 [Info.plist](../../../app/ios/Runner/Info.plist)。Android 对应是 Manifest 的 `<uses-permission CAMERA>`。
- **为什么切主线程：** `requestAccess` 的完成回调**在任意（非主）线程**执行，而 **UIKit 的 `present` 必须在主线程**调用。不切线程直接 present 属于"后台线程碰 UI"，行为未定义/可能崩。所以 `DispatchQueue.main.async { presentScanner(...) }`——这跟 L1「耗时活挪后台、结果回主线程」、L3「sink 回主线程投递」是同一条线程纪律。

## Q4. 原生扫码页 present 出去后，Flutter 里 `await ScanBridge.scan()` 处于什么状态？Flutter 页还在不在栈里？什么时候才返回？

- `await ScanBridge.scan()` 处于**挂起（pending）**状态——它在等原生那次延迟的 `result`，对应的 `Future` 还没完成。`_startScan` 里 `_scanning=true`、按钮被禁用，就是这段挂起期的 UI 表现。
- **Flutter 页仍在 widget 栈里**，只是被原生扫码页**盖在上面**（原生 present 是盖一层，不是销毁 Flutter 页）。所以原生页 dismiss 后，下面的 L4 页原样还在，`setState` 能直接更新它。
- **什么时候返回：** 用户在原生页点了"扫到 xx"或"取消" → 原生 `dismiss` → dismiss 的 completion 里调 `pendingResult?(code)` → Flutter 侧 `Future` 完成 → `await` 返回 `ScanOutcome`，`setState` 刷新结果卡片。见 [l4_scan_page.dart](../../../app/lib/lessons/l4/l4_scan_page.dart) 的 `_startScan`。

## 附一：课后练习核对（hint 入参：打开原生页时携带参数）

要求给 `scan()` 加入参 `hint`，透传给原生扫码页当标题。核对点：

| 端 | 实现 | 位置 |
|---|---|---|
| Dart 桥 | `scan({String? hint})` → `invokeMethod('scan', {'hint': hint})` | [scan_bridge.dart](../../../app/lib/lessons/l4/scan_bridge.dart) |
| Dart 页面 | 调用时带 `scan(hint: '请对准商品条码 / 优惠券码')` | [l4_scan_page.dart](../../../app/lib/lessons/l4/l4_scan_page.dart) |
| Swift | `call.arguments['hint']` → 一路透传到 `MockScannerViewController(hint:)`，`title = hint ?? 默认` | [AppDelegate.swift](../../../app/ios/Runner/AppDelegate.swift) |
| Kotlin | `(call.arguments as? Map<*,*>)?.get("hint")` → 暂存 `pendingHint` → `Intent.putExtra("hint", ...)`；`ScannerActivity` `title = intent.getStringExtra("hint")` | [MainActivity.kt](../../../app/android/app/src/main/kotlin/com/wenbo/native_lab/MainActivity.kt) / [ScannerActivity.kt](../../../app/android/app/src/main/kotlin/com/wenbo/native_lab/ScannerActivity.kt) |
| 测试 | 断言 `call.arguments['hint']` 收到透传值 + 不传时为 null | [l4_scan_test.dart](../../../app/test/l4_scan_test.dart) |

**要点：** "打开原生页"和 L1/L2 的"调方法"一样能带 Map 入参下去——codec 规则完全一致。Android 侧因为要跨 Activity，hint 也得像 `pendingResult` 一样暂存成 `pendingHint`（结果延迟、参数也得留住）。
