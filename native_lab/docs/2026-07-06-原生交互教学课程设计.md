# NativeLab：Flutter 与原生交互教学课程设计

> 日期：2026-07-06
> 状态：设计已确认，待写实施计划
> 学员背景：高级 iOS 工程师（Swift/OC、UIKit/SwiftUI 熟练），已通过 WanShop 项目掌握 Flutter 基础（Widget/布局/Riverpod/go_router/dio/测试），见 `wan_android/`。

## 1. 目标与非目标

**目标**：新建一个独立教学项目，系统学习 Flutter 与原生（iOS 为主、Android 对照）的交互，例子全部贴合企业真实需求，最终达到"在公司能独立承接混合开发需求"的水平。

**非目标**：

- FFI（dart:ffi 调 C/C++）不进大纲，以后有需要再单开。
- 原生侧不上 XCTest/单元测试，以模拟器手动验证为主（避免节奏拖慢）。
- Android 侧不强制每课跑模拟器（受构建环境限制），但 Kotlin 对照实现和讲解每课必有。

## 2. 已确认的决策

| 决策点 | 结论 |
|---|---|
| 平台覆盖 | iOS(Swift) 为主线、iPhone 模拟器验证；Android(Kotlin) 每课给完整对照实现与讲解，不强制跑安卓模拟器 |
| 课程范围 | 核心频道通信（MethodChannel/EventChannel/BasicMessageChannel、编解码、Pigeon、PlatformView）+ 插件开发 + add-to-app 混合开发 |
| 工程组织 | 方案 A：一个「原生能力实验室」主 App 逐课生长，插件课抽包、add-to-app 课建原生壳，最终形成企业真实的三工程 monorepo 形态 |

## 4. 课程大纲（10 课时，L0–L9）

难度循序渐进：从"读一个值"到"改造整个原生 App"。每课例子对应一个企业真实需求。

| 课时 | 主题 | 企业真实场景 | 新知识点 |
|---|---|---|---|
| L0 | 工程创建与原生工程解剖 | 入职第一天：看懂 Flutter 工程里 ios/ 和 android/ 到底是什么 | Runner 工程结构、FlutterViewController、引擎与线程模型概念铺垫 |
| L1 | MethodChannel：Flutter 调原生 | 设备信息上报（机型/系统版本/电池，风控和日志的标配需求） | MethodChannel、result.success/error/notImplemented、回调线程、channel 的 Dart 侧单测 |
| L2 | 数据编解码与复杂参数 | 埋点桥：Flutter 埋点统一走原生已接好的统计 SDK（事件名 + 属性 Map） | StandardCodec 类型映射表（Dart↔Swift↔Kotlin）、Map/List/二进制传参 |
| L3 | EventChannel：原生持续推流 | 网络状态监听（弱网提示、断网重连，电商 App 必备） | EventChannel=Stream、onListen/onCancel、对比 NotificationCenter/Combine |
| L4 | 页面级混合 + 权限 | Flutter 页跳原生扫码页（present 原生 VC），扫码结果回传 | 原生页面打开/关闭、相机权限申请、双向回传 |
| L5 | Pigeon 类型安全生成 | 团队协作痛点：手写 channel 全是字符串魔法值，重构 L1/L2 为 Pigeon | Pigeon 定义文件、代码生成、和手写 channel 的对比 |
| L6 | PlatformView：视图级混合 | 页面里嵌原生地图（MKMapView，国内高德/百度 SDK 同理） | UiKitView、创建参数传递、性能代价、和页面级混合怎么选 |
| L7 | 插件开发 | 把 L1 设备信息桥抽成独立 plugin 包（企业"自家插件库"的诞生过程） | plugin 模板、Dart API 层设计、path 依赖、联邦插件概念 |
| L8 | add-to-app：原生工程接入 Flutter | 公司已有原生 App 要上 Flutter（建原生壳模拟），CocoaPods 接入模块 | flutter module、pod 集成、FlutterViewController 打开 Flutter 页 |
| L9 | add-to-app：引擎管理与通信 | 混合 App 的工程化：引擎预热、原生↔模块双向通信、路由协调 | FlutterEngine 生命周期、多引擎取舍、混合路由栈实践 |

平台覆盖说明：L1–L7 每课 Swift 主讲 + Kotlin 完整对照；L8–L9 是 iOS 专属场景（原生壳即 iOS 工程），Android 侧只讲概念差异。

## 5. 课时文档模板

每课在 `native_lab/docs/lessons/` 下一份 `LXX-主题.md`，结构沿用 WanShop M10 文档：

1. **本课要掌握什么**——核心概念 + 为什么企业需要它
2. **控件/API 速查表**（硬性规则）——本课新出现的 Flutter API 和原生 API，逐个给 ① iOS 类比 ② 用法 + 关键参数 ③ 易踩的坑，表格呈现
3. **代码地图**——本课动了哪些文件、各自职责，Dart 侧和原生侧分开列
4. **双端对照**——Swift 和 Kotlin 实现的差异点（L8/L9 改为"iOS 专属说明"）
5. **自测清单**——若干个"你能回答/能做到吗"的检查项
6. **课后练习**——一个小改动题，动手才算学会

## 6. 进度门禁（学完一课才能进下一课）

三层机制：

- **`docs/lessons/README.md` 进度表**：每课一行，状态 ⬜ 未开始 / 🔵 进行中 / ✅ 已完成。这是唯一进度真身。
- **过关条件**：代码在 iPhone 模拟器跑通（截图验证）+ `flutter analyze` 0 issue + 全量 `flutter test` 通过 + 学员口头确认"过了"。确认前不开下一课。
- **App 内课程列表**同步点亮已完成课时——进度表的可视化副本。实现上就是 Dart 里一份硬编码的课程注册表（每课的标题/路由/状态），每课过关提交时把状态改为已完成，不做持久化，避免为进度显示引入额外复杂度。

## 7. 每课教学流程

概念讲解（iOS 类比先行）→ 写 Dart 侧 → 写 Swift 侧 → iPhone 模拟器跑通、截图验证 → Kotlin 对照讲解 → 写课时文档 → 学员过自测清单 → 学员确认后 git 提交。

git 约定：直接提交 main，中文提交信息（如 `L1 设备信息桥：MethodChannel 入门`），只 `git add native_lab/`，提交前等学员确认。

## 8. 测试策略

- Dart 侧每课带单测：用 `TestDefaultBinaryMessenger.setMockMethodCallHandler` mock 原生响应，不起模拟器即可测 channel 逻辑。该技能本身是 L1 教学内容之一，也是企业面试高频点。
- 改动后跑**全量** `flutter test`，不只跑新增文件（WanShop M7 的教训）。
- 原生侧以模拟器手动验证为主。

## 9. 错误处理教学线（贯穿暗线）

channel 调用的三类失败在 L1 引入，之后每课的例子都按企业标准处理，不写裸调用：

1. 原生抛 `PlatformException`（对应 result.error）
2. 方法未实现 `MissingPluginException`（对应 notImplemented，热重启后原生侧未注册也会触发）
3. channel 本身无超时机制，需要时自己用 `Future.timeout` 包
