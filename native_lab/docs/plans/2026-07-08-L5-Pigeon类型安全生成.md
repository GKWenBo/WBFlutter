# NativeLab L5：Pigeon 类型安全生成 实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **教学项目特别说明**：本计划的"执行"是一堂课——边写边讲（iOS 类比先行）。本课核心是"**代码生成**"：先写一份 Pigeon 定义文件，用 `dart run pigeon` 一键生成三端（Dart/Swift/Kotlin）的强类型胶水代码，再各自填原生实现。真机验证需学员在模拟器上操作（进 L5 → 看设备信息 Pigeon 版 → 开/关电量订阅看反向推流）。

**Goal:** 前四课都在**手写 `MethodChannel`/`EventChannel`**——channel 名是字符串魔值、参数靠 `Map` 手拼手拆、类型对不上要到**运行期**才崩（`TypeError`/`as Int` 崩）。本课引入 **Pigeon**：**写一份 Dart 定义文件当"契约"，代码生成器产出三端类型安全的胶水**，把"字符串魔值 + 手写编解码 + 运行期崩"换成"**定义即契约 + 生成代码 + 编译期红线**"。这是团队协作的刚需——契约集中一处、改一处三端同步、重命名/漏字段编译就报。

**场景（企业真实）：** 把 **L1 的设备信息桥**用 Pigeon 重写成 **L5 独立页**，**L1 手写版原样保留**，两页并排 = 活教材（手写 vs 生成）。同时用 **FlutterApi**（原生→Flutter 的类型安全反向）推**电量变化**，与 **L3 的 EventChannel** 做「同一件事、两种做法」对比：L3 是字符串通道 + 手拆 Map 的推流，L5 是生成的强类型回调。

**Architecture:** 新增一层——**契约层 `pigeons/device_info_api.dart`**（唯一手写的源头），生成物落 `lib/lessons/l5/messages.g.dart`（Dart）、`ios/Runner/DeviceInfoMessages.g.swift`、`android/.../DeviceInfoMessages.g.kt`，外加 `test/messages_test.g.dart`（Pigeon 生成的测试桩）。Dart 侧 `DeviceInfoPigeonBridge` 薄封装生成的 `DeviceInfoHostApi()` + 实现生成的 `DeviceEventFlutterApi` 把电量回调转成 `Stream`；`L5PigeonPage`（`StatefulWidget` + `setState`，够用）。iOS/Android 各实现生成的 HostApi 协议并持有 FlutterApi 实例回推。**注意：Pigeon 不再需要我们手写 channel 名和 codec——它自己生成一条私有 channel + 自定义 codec。**

**★ 与前四课的关键差异（本课核心心智）：**
- L1–L4 是"**手写通道**"：`MethodChannel('字符串')` + `invokeMethod('方法名', Map)`，方法名/参数键/类型全是**约定**，两端各写一遍，错了运行期才知道。Pigeon 是"**生成通道**"：契约写一次，`getDeviceInfo()` 是**真方法**（改名、少字段、类型错 → 编译报错），参数/返回是**生成的类**（不是 `Map`）。
- Pigeon **不取代** MethodChannel，是**盖在它上面的代码生成器**——底层仍是二进制 channel，只是 codec 和收发样板由它生成。所以 L1–L4 的心智（异步、错误传播）全都还在，只是不用手拼了。
- 反向：`@FlutterApi()` = 原生主动调 Flutter 的类型安全版，对照 **L3 EventChannel**。区别：EventChannel 是"一条 Stream、事件是裸值/Map"；FlutterApi 是"生成的方法回调、参数是强类型类"。本课把两者摆一起讲取舍。

**Tech Stack:** `pigeon`（dev_dependencies）；`@ConfigurePigeon` / `@HostApi` / `@FlutterApi` / `@async` 注解；`dart run pigeon` 代码生成；iOS 实现生成的 `DeviceInfoHostApi` 协议 + `DeviceInfoHostApiSetup.setUp` + 持 `DeviceEventFlutterApi`，电量用 `UIDevice.isBatteryMonitoringEnabled` + `batteryLevelDidChangeNotification`；Android 实现生成接口 + `DeviceInfoHostApi.setUp` + `BroadcastReceiver(ACTION_BATTERY_CHANGED)`。测试用 **Pigeon 生成的 `TestDeviceInfoHostApi`**（`dartTestOut`）当 mock host——本课的新测试姿势：桩由 Pigeon 生成，不再手写 `setMockMethodCallHandler`。

// L5 Pigeon 契约：本课唯一手写的"源头"。写完跑 `dart run pigeon
// --input pigeons/device_info_api.dart` 生成三端胶水。这个文件本身不参与运行——
// 它只是"给代码生成器看的规格说明"（对照：iOS 里你不会手写 URLSession 的样板，
// 而是描述接口让工具产出；这里描述 channel 契约让 Pigeon 产出收发代码）。
import 'package:pigeon/pigeon.dart';

// @ConfigurePigeon：把"生成到哪、包名是什么"钉在契约文件里，
// 这样 `dart run pigeon` 不用一长串命令行参数。
@ConfigurePigeon(PigeonOptions(
  dartOut: 'lib/lessons/l5/messages.g.dart',
  // dartTestOut：额外生成"测试用的 mock host"（TestDeviceInfoHostApi）——
  // 这就是本课不再手写 setMockMethodCallHandler 的原因：桩由 Pigeon 生成。
  dartTestOut: 'test/messages_test.g.dart',
  swiftOut: 'ios/Runner/DeviceInfoMessages.g.swift',
  kotlinOut:
      'android/app/src/main/kotlin/com/wenbo/native_lab/DeviceInfoMessages.g.kt',
  kotlinOptions: KotlinOptions(package: 'com.wenbo.native_lab'),
  dartPackageName: 'native_lab',
))

// 数据类：对照 L1 手写时靠 Map{'model': ..., 'systemVersion': ...} 手拼手拆，
// 这里声明成"字段强类型的类"，Pigeon 生成三端等价的类 + 编解码。
// 字段非空 → 三端生成的都是非空类型，少传一个字段编译期就报（手写 Map 做不到）。
class PigeonDeviceInfo {
  PigeonDeviceInfo({
    required this.model,
    required this.systemName,
    required this.systemVersion,
    required this.isPhysicalDevice,
    required this.batteryLevel, // 电量可能取不到 → 允许空
  });
  String model;
  String systemName;
  String systemVersion;
  bool isPhysicalDevice;
  int? batteryLevel;
}

class BatteryInfo {
  BatteryInfo({required this.level, required this.isCharging});
  int level; // 0–100，-1 表示未知
  bool isCharging;
}

// @HostApi：Flutter → 原生（对照 L1 的"Dart 调原生"）。
// 这些方法名就是契约——生成后 Dart 侧 host.getDeviceInfo() 是真方法，
// 改名/改签名两端一起编译报错（对照手写：invokeMethod('getDeviceInfo') 打错字运行期才崩）。
@HostApi()
abstract class DeviceInfoHostApi {
  // @async：原生实现端拿到一个 completion 回调，可异步完成（演示 Pigeon 的异步契约）。
  @async
  PigeonDeviceInfo getDeviceInfo();

  // 开始/停止电量订阅——对照 L3 EventChannel 的 onListen/onCancel，
  // 但这里是普通方法调用，推流走下面的 FlutterApi。
  void startBatteryUpdates();
  void stopBatteryUpdates();
}

// @FlutterApi：原生 → Flutter（反向）。生成后原生侧持有一个
// DeviceEventFlutterApi 实例，调 onBatteryChanged(info) 就把强类型对象推给 Dart。
// 这是 L3 EventChannel 的"类型安全版"：L3 事件是裸 Map 要手拆，这里是生成的 BatteryInfo。
@FlutterApi()
abstract class DeviceEventFlutterApi {
  void onBatteryChanged(BatteryInfo info);
}
```

import 'dart:async';

import 'messages.g.dart';

/// L5 设备信息桥（Pigeon 版）。对照 L1 的手写 DeviceInfoBridge：
/// - L1：手写 MethodChannel('com.wenbo.native_lab/device_info') + invokeMethod('...')
///   + 手拆返回的 Map 塞进自己写的 DeviceInfo 模型。
/// - L5：直接调生成的 DeviceInfoHostApi().getDeviceInfo()，返回生成的强类型
///   PigeonDeviceInfo——channel 名、codec、Map 拆装全没了，交给生成代码。
///
/// 这层薄封装做两件事：①持有生成的 HostApi 调用端；②实现生成的
/// DeviceEventFlutterApi 接收端，把原生回推的电量事件转成 Stream（对照 L3 用法）。
class DeviceInfoPigeonBridge implements DeviceEventFlutterApi {
  DeviceInfoPigeonBridge() {
    // 注册"我"为反向回调的接收者（对照 L3 里 EventChannel 的 receiveBroadcastStream，
    // 但这里是把生成的接口实现挂到 Pigeon 的私有 channel 上）。
    DeviceEventFlutterApi.setUp(this);
  }

  final DeviceInfoHostApi _host = DeviceInfoHostApi();
  final StreamController<BatteryInfo> _battery =
      StreamController<BatteryInfo>.broadcast();

  /// Flutter→原生：取设备信息（生成方法，编译期类型安全）。
  Future<PigeonDeviceInfo> getDeviceInfo() => _host.getDeviceInfo();

  /// 开/停电量订阅（对照 L3 onListen/onCancel，但走普通方法调用）。
  Future<void> startBatteryUpdates() => _host.startBatteryUpdates();
  Future<void> stopBatteryUpdates() => _host.stopBatteryUpdates();

  /// 原生回推的电量事件流（对照 L3 的 statusStream）。
  Stream<BatteryInfo> get batteryStream => _battery.stream;

  /// 生成接口的回调实现——原生每调一次 onBatteryChanged，就进这里。
  @override
  void onBatteryChanged(BatteryInfo info) => _battery.add(info);
}
```

```swift
/// L5 设备信息桥（原生侧，Pigeon 版）。对照 L1 的手写 DeviceInfoBridge：
/// L1 我们手写 FlutterMethodChannel + setMethodCallHandler + 手拼返回字典；
/// 这里实现 Pigeon 生成的 DeviceInfoHostApi 协议——方法签名、参数类型都由生成代码定死，
/// 少实现一个方法 / 类型不对 → 编译报错（手写通道做不到）。
final class DeviceInfoPigeonHost: NSObject, DeviceInfoHostApi {
  // 反向推流用：持有生成的 FlutterApi 调用端（对照 L3 的 sink，但这是强类型方法）。
  private let flutterApi: DeviceEventFlutterApi
  private var observing = false

  init(binaryMessenger: FlutterBinaryMessenger) {
    self.flutterApi = DeviceEventFlutterApi(binaryMessenger: binaryMessenger)
    super.init()
  }

  // @async 契约 → 生成的是带 completion 的签名（对照 L1 handler 里的 result(...)）。
  func getDeviceInfo(completion: @escaping (Result<PigeonDeviceInfo, Error>) -> Void) {
    let d = UIDevice.current
    d.isBatteryMonitoringEnabled = true
    let level = d.batteryLevel < 0 ? nil : Int64(d.batteryLevel * 100)
    completion(.success(PigeonDeviceInfo(
      model: d.model,
      systemName: d.systemName,
      systemVersion: d.systemVersion,
      isPhysicalDevice: TARGET_OS_SIMULATOR == 0,
      batteryLevel: level)))
  }

  func startBatteryUpdates() throws {
    guard !observing else { return }
    observing = true
    UIDevice.current.isBatteryMonitoringEnabled = true
    // 对照 L3：注册通知观察（≈ EventChannel onListen）。
    NotificationCenter.default.addObserver(
      self, selector: #selector(batteryChanged),
      name: UIDevice.batteryLevelDidChangeNotification, object: nil)
    NotificationCenter.default.addObserver(
      self, selector: #selector(batteryChanged),
      name: UIDevice.batteryStateDidChangeNotification, object: nil)
    batteryChanged() // 立即推一次当前值
  }

  func stopBatteryUpdates() throws {
    observing = false
    NotificationCenter.default.removeObserver(self) // ≈ EventChannel onCancel
  }

  @objc private func batteryChanged() {
    let d = UIDevice.current
    let info = BatteryInfo(
      level: d.batteryLevel < 0 ? -1 : Int64(d.batteryLevel * 100),
      isCharging: d.batteryState == .charging || d.batteryState == .full)
    // 通知回调在主线程，直接调；生成的 FlutterApi 帮我们编码 + 过 channel。
    flutterApi.onBatteryChanged(info: info) { _ in }
  }
}
```
