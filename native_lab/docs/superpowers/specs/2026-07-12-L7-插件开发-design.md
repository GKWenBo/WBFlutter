# L7 设计：插件开发（抽出 nl_device_kit）

- 状态：已定稿（brainstorming 通过）
- 日期：2026-07-12

## 一句话

把 L1 的设备信息桥抽成独立 Flutter 插件包 `plugins/nl_device_kit`（官方模板三层结构），
app 通过 `path:` 依赖它、用新建的 L7 页消费它，讲透"把能力打包成可复用插件"这件工程事：
插件工程结构、Dart API 层设计、自动注册、path 依赖、联邦插件概念。

## 决策记录（brainstorming 结论）

1. **插件结构 = ② 官方模板默认三层**（单包，内部分 app-facing / platform-interface / method-channel 三层）。
   完整联邦多包（③）留作讲义延伸，不实现。
2. **A. 并行保留**：L1 手写版原样不动，新建 L7 页用插件取同样信息（沿用 L5/L6 对照套路）。
   - 插件用独立 channel 名 `nl_device_kit`，与 L1 的 `com.wenbo.native_lab/device_info` 互不干扰。
   - 插件装 L1 的**全部三个能力**：`getDeviceInfo` / `getBatteryLevel` / `getUptime`。

## 插件结构

```
plugins/nl_device_kit/
├── lib/
│   ├── nl_device_kit.dart                      # app 面向公开 API（NlDeviceKit 类）
│   ├── nl_device_kit_platform_interface.dart   # 抽象接口（PlatformInterface，联邦接缝）
│   ├── nl_device_kit_method_channel.dart       # MethodChannel 实现（默认 instance）
│   └── src/device_info.dart                    # 插件自己的 DeviceInfo 强类型模型
├── ios/Classes/NlDeviceKitPlugin.swift         # FlutterPlugin：注册 + 三方法
├── android/.../NlDeviceKitPlugin.kt            # FlutterPlugin：注册 + 三方法
├── pubspec.yaml                                # flutter: plugin: platforms: {ios, android}
└── test/                                        # 插件包自己的测试
```

- 公开 API：`NlDeviceKit().getDeviceInfo() / getBatteryLevel() / getUptime()`，
  内部委托 `NlDeviceKitPlatform.instance`（默认是 `MethodChannelNlDeviceKit`）。
- `DeviceInfo`（model/systemName/systemVersion/appVersion）是插件自己的类型，
  与 L1 的 `l1/device_info.dart` 各自独立（并行对照，互不引用）。
- 原生包名/组织：`com.wenbo.nl_device_kit`。channel 名：`nl_device_kit`。

## 关键认知差异（写进讲义）

- **自动注册 vs 手写注册**：前 6 课原生桥都在 `AppDelegate`/`MainActivity` **手写注册**；
  插件的原生代码由 **GeneratedPluginRegistrant 自动注册**（实现 `FlutterPlugin` 的
  `onAttachedToEngine`/iOS `register(with:)`）。这是"插件"与"app 内嵌桥"最大的工程差异。
- **path 依赖**：app 的 `pubspec.yaml` 用 `nl_device_kit: { path: ../plugins/nl_device_kit }`，
  企业"自家插件库"的雏形（真发布则改 git/pub.dev 依赖）。
- **platform-interface 接缝 = 联邦概念的锚点**：为什么多一层抽象接口——为"同一 API、多种实现"
  （不同平台 / 不同后端 / mock）留缝，也是测试价值所在。完整联邦多包（platform_interface 包 +
  app-facing 包 + 各平台实现包）留作讲义"再往上一层"的延伸。

## 代码地图

**新增插件包 `plugins/nl_device_kit/`**
- `lib/nl_device_kit.dart`：`NlDeviceKit` 类，三方法委托 platform instance。
- `lib/nl_device_kit_platform_interface.dart`：`NlDeviceKitPlatform extends PlatformInterface`，三抽象方法 + 默认 instance。
- `lib/nl_device_kit_method_channel.dart`：`MethodChannelNlDeviceKit`，`MethodChannel('nl_device_kit')` 实现三方法。
- `lib/src/device_info.dart`：`DeviceInfo` 模型 + `fromMap`。

- `android/.../NlDeviceKitPlugin.kt`：`FlutterPlugin + MethodCallHandler`，`onAttachedToEngine` 挂 channel（搬 L1 的 handler 逻辑）。

## 课后练习

给插件加一个新能力（如 `getDeviceModelName()`），走一遍完整回路：
platform interface 加抽象方法 → method channel 实现加方法 → iOS/Android 两端各加实现 →
`NlDeviceKit` 暴露 → app 调用 → 补一条 mock platform 的测试。体会插件的 API 演进与"改一处、
接口/实现/调用点一起动"的约束。
