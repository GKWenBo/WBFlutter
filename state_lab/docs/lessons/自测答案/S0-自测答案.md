# S0 自测答案（对照版）

> 对应 [S0-工程起步与setState基线.md](../S0-工程起步与setState基线.md) 第五节自测清单。先自己写完再看。

## 1. 详情页加购后，为什么读 `widget.cart` 拿得到最新数据、界面却不自己刷新？

**读靠引用，刷新靠通知——这是两件事。** 所有页面持有的 `cart` 是同一个可变 List 引用，所以任何时刻读它都是最新值。但 Flutter 不做数据观察：改 List 不会触发任何 rebuild，界面只在 setState / 父级重建时才刷新。

类比 iOS：多个 VC 共享同一个 `NSMutableArray`，改完不发 notification，别的 VC 的 tableView 不会自己 `reloadData`。"数据是对的"和"界面知道数据变了"是两回事——后者在 v0 里全靠手动。

## 2. 「双重 setState」两次各刷新谁？漏掉会怎样？

- **根那次**（`widget.onAddToCart` 内部触发）：重建 `V0ShopRoot` 的 build 子树——即**列表页**，让列表页 AppBar 角标更新。
- **本页那次**（`setState(() {})`）：重建**详情页自己**，让本页角标 +1。

漏掉根那次：当场看不出问题，**回到列表页才发现角标没动**（陈旧 UI 的延迟暴露，最难排查的一类）。漏掉本页那次：点"加入购物车"后本页角标纹丝不动，看起来"没加上"，但数据其实已经进了 cart——重进页面就"好了"，典型的"玄学 bug"。

## 3. 为什么合计金额用 getter 现算，不存 `_totalPrice`？

存变量 = 制造第二份事实。cart 和 `_totalPrice` 必须在**每一个**改动点（加购、增减、删除、清空）同步更新，漏一处就不一致，而且不一致是静默的。getter 现算保证**单一事实来源**：cart 对，合计就一定对。build 时算一次 fold，量级几十个元素，性能不构成理由。

类比 iOS：SwiftUI 里写 computed property，而不是再开一个 `@State` 手动同步。

## 4. `mounted` 在防什么？对应 iOS 什么习惯？

防 **await 回来时 State 已销毁**：用户在请求返回前就 pop 了页面，此时调 setState 抛 `setState() called after dispose()`。`if (!mounted) return;` 就是"确认自己还活着再动 UI"。

对应 iOS 闭包里的 `[weak self]` + `guard let self else { return }`——异步回调持有的 self 可能已经没了。

## 5. `_requestSeq` 防什么场景？不做会怎样？

防**慢请求覆盖新结果**。连续输入 "ph" → "phone" 发出两枪，如果 "ph" 的响应更慢、后到，不丢弃它就会把 "phone" 的结果整个盖掉——用户看到的列表和输入框对不上。序号机制只允许"最新一发"落地，就是手搓版 Combine `switchToLatest` / RxSwift `flatMapLatest`。

## 6. `RefreshIndicator.onRefresh` 为什么必须返回 `Future`？

**转圈动画的生命周期就绑在这个 Future 上**：松手后 spinner 一直转，直到 Future 完成才收起。这是框架替你对齐"动画状态"和"异步任务状态"。如果返回一个立刻完成的 Future（比如忘了 await），圈会瞬间消失但数据还没回来——用户以为刷新完了。

## 7. `Dismissible` 的 `onDismissed` 不立刻删数据会怎样？key 为什么不能用 index？

滑动动画结束后该项已从视觉上移除，但下一帧 build 时数据还在、同一个 key 的 Dismissible 又出现在树里，Flutter 直接红屏：*"A dismissed Dismissible widget is still part of the tree."* 所以 `onDismissed` 里必须同步把数据删掉。

key 用 index 的话：删掉第 i 项后，原第 i+1 项顶上来继承了同一个 key/Element/State，Flutter 认为"还是原来那个 Widget"，状态串位。用业务 id（`ValueKey(product.id)`）才能让框架正确对齐"哪条数据对应哪个元素"。类比 iOS：`deleteRows` 和数据源必须同帧一致，否则 `NSInternalInconsistencyException`。

## 8. 首页点 v1 为什么弹 SnackBar 而不崩溃？门禁看什么字段？

判断依据是注册表里的 **`builder` 字段**：`unlocked => builder != null`。onTap 先查门禁——`builder == null` 走 SnackBar 分支提示解锁课次，非空才 `push(builder(context))`。不会崩溃是因为压根不会走到"调用 null"那一步：**用类型系统（可空性）表达业务状态**，而不是靠运行时兜底。

## 9. （你抓到的 bug）购物车清空后返回详情页，角标为什么不清零？

三方各自的重建范围：

- **根 setState**：只重建根的 build 子树 = 列表页。够不着详情页（它是 push 到 Navigator 上的兄弟路由）。
- **购物车页 setState**：只重建购物车页自己。
- **详情页**：pop 回来时 Flutter **不会重建下层路由**——没有 viewWillAppear 这种"我要露脸了"的回调。

数据其实是对的（共享引用），但没有任何人通知详情页重新 build——它显示的还是上次 build 的快照。

**v0 的修法**：push 时 `await` 路由的 Future，pop 回来后 `if (!mounted) return; setState(() {})`。对应 iOS 在 `viewWillAppear` / push completion 里 `reloadData` 的习惯。代价是**每一条"深层页面可能改数据"的导航都得这么包**，漏一处就是一处陈旧 UI——这个不可维护性正是 S1 InheritedWidget/依赖注册机制要解决的问题。
