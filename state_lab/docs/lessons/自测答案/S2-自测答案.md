# S2 自测答案（对照版）

> 对应 [S2-Provider版MiniShop.md](../S2-Provider版MiniShop.md) 第五节自测清单。附录：课后练习实测记录。

## 1. `watch` 在事件回调里调会发生什么？为什么框架要拦？

直接抛错：*"Tried to use `context.watch` outside of the `build` method"*（Provider 在断言层拦下）。

根因是 S1 讲过的框架级约束：watch 底下是 `dependOnInheritedWidgetOfExactType`，依赖登记必须发生在 Element 的 build 阶段——框架靠"谁在 build 谁登记"保证依赖名单和树结构一致。回调不属于任何一次 build，登记了也没有归属：下次树重排时这条依赖既没法校验也没法清理，只会变成幽灵订阅。所以规则不是 Provider 的洁癖，是 InheritedWidget 机制的边界。回调里改用 `read`（只取值、不登记）。

## 2. `create` 和 `.value` 的唯一本质区别？两种用反各导致什么事故？

唯一本质区别：**生命周期归属**。`create` = 这个对象由 provider 生、由 provider 销（卸载时自动 `dispose`）；`.value` = 对象的生死别人在管，provider 只负责挂树。其余行为（查找、订阅、通知）完全一致。

- **`.value` 包新建对象**：`ChangeNotifierProvider.value(value: CartModel())`——没有任何人会 dispose 它，监听者名单常驻，内存泄漏；如果里面还有 Timer/StreamSubscription，就是幽灵任务。
- **`create` 包已有对象**：`create: (_) => existingModel`——provider 卸载时把别人的对象 dispose 了，原主人下次调用就撞 *"A CartModel was used after being disposed"*。

v1 里的用法对照：根上新建 CartModel → `create`；每条 push 复用同一个 CartModel → `.value`。

## 3. `Selector` 判定重不重建的完整流程？`(c) => c.items` 错在哪？

流程：`notifyListeners` 开火 → provider element 通知依赖者 → **先跑 selector 算出新选值** → 与上次的选值比 `==` → 不等才 `markNeedsBuild` 重建 builder。通知只是"敲门"，重建与否由选值判等决定——这就是"字段级粒度"的全部实现。

`(c) => c.items` 的两种死法：

- 模型直接暴露内部 List（同一个可变引用）：引用从没变过，`==` 恒真 → **永不重建**，界面死数据；
- 我们的 CartModel 每次 getter 返回新的 `List.unmodifiable` 拷贝：引用次次不同，`==` 恒假 → **次次重建**，Selector 退化成 Consumer。

正解：选**标量**（`totalCount`、`items.length`）或实现了值语义 `==` 的不可变对象（S3 的 Equatable 就是干这个的——伏笔）。

## 4. 页面级 provider 的模型什么时候 create、什么时候 dispose？lazy 默认值带来什么行为？

- **create**：默认 `lazy: true`——不在挂树时执行，而在**第一次被 read/watch** 时执行。v1 列表页的模型之所以首帧就创建并 `loadFirst()`，是因为 body 的 `Consumer` 首帧就 watch 了它；如果首帧没人消费，创建（和请求）会推迟到第一个消费者出现。需要"挂树即创建"就传 `lazy: false`。
- **dispose**：provider element 卸载时（页面 pop、路由销毁）自动调用——SearchModel 里的防抖 Timer 也在 dispose 里一起 cancel，托管链条闭环。

对照 v0：initState 手动建、dispose 手动销、忘一头就泄漏——现在两头都不用记。

## 5. `_disposed` 守卫防什么？对应页面侧的什么？没有它会看到什么？

防**dispose 后的迟到通知**：页面 pop → provider 托管 dispose 模型 → 在途的网络请求这时才回来 → 回调里 `notifyListeners()` → 崩：*"A ProductListModel was used after being disposed."*

对应关系：页面侧 `if (!mounted) return;` 防的是 **State 死了还 setState**；模型侧 `if (_disposed) return;` 防的是**模型死了还 notify**。同一种异步病（回调活得比宿主长），两处疫苗。iOS 类比不变：`[weak self]` + `guard let self`。

## 6. 搜索页为什么连 api 参数都不用传了？`Provider<T>` 和 `ChangeNotifierProvider<T>` 差在哪？

因为 ProductApi 也上了树：v1 根的 `MultiProvider` 里挂了 `Provider<ProductApi>`，push 搜索页时随 `.value` 一起 re-provide，SearchModel 在页面级 create 里 `context.read<ProductApi>()` 就地取用——**服务对象走 DI 树，构造函数只剩业务参数**（详情页的 product）。

差别：`Provider<T>` 是纯 DI 货架——不要求 T 可监听、不订阅、不通知，适合服务/配置/客户端这类"不变的依赖"；`ChangeNotifierProvider<T>` 在此之上加了"订阅 T 的通知并扩散给依赖者 + 托管 dispose"。对 ProductApi 用后者是浪费，对 CartModel 用前者是失灵（改了没人知道）。

## 7. v1 里哪些依赖是字段级、哪些是页面级？为什么？

- **字段级**：三处角标——列表页 `Selector`、详情页 `Builder + context.select`、搜索页 `Selector`，都只选 `totalCount`。理由：角标只关心一个数字，购物车内容再怎么翻江倒海，数字不变就不该有任何重建（实测见附录）。
- **页面级**：购物车页 `context.watch<CartModel>()`——整页每一寸都在展示 cart（列表、合计、清空按钮的可用态），页面级就是它的"最小充分粒度"，再细分是过度设计；列表/搜索 body 的 `Consumer<页面模型>` 同理。

口诀落地：**粒度选"够用的最小"，不是"越细越好"**——细粒度的代价是代码碎片化，收益只在"高频通知 + 大子树"处显著。

## 8. `Consumer` 的 child 参数解决什么问题？

builder 里通常有一部分子树**根本不依赖 model**（静态装饰、大图、复杂布局）。不用 child 时它们跟着 builder 每次重跑；把它们作为 `child:` 传入，Consumer 会把同一个 widget 实例原样塞回 builder 的第三参——实例相同,Element 层直接跳过重建（S1 讲过的"child 相同实例 → 子树不重建"）。本质是把"变的"和"不变的"在构造期就分家。

## 9. MultiProvider 里 provider 的顺序什么时候有意义？

当后面的 provider 在 `create`/`update` 里要消费前面的 provider 时——MultiProvider 只是嵌套的语法糖，展开后排在后面的是排在前面的**子孙**，子孙才能 read 祖先。v1 根上 `Provider<ProductApi>` 必须排在 `ChangeNotifierProvider<CartModel>` 前面吗？不必须（CartModel 不依赖 api）；但如果上了 `ProxyProvider<Auth, ApiClient>`，Auth 必须在前。互不依赖时顺序随意，有依赖时"被依赖者在前"。

## 10. Provider 的两个上限（Riverpod 的立项理由）？

1. **运行时才安全**：provider 忘挂/类型没对上，编译器毫无察觉，跑到那行才抛 `ProviderNotFoundException`。类型查找也意味着同一类型只能挂一份（想要两个 `Provider<String>` 就得包装类型）。
2. **组合与脱树困难**：状态间依赖靠 `ProxyProvider` 手拼，链一长就是嵌套地狱；一切读取都绑着 `BuildContext`，逻辑层想脱离 Widget 树使用/测试就别扭（v1 的模型测试干净，是因为我们把逻辑挤进了纯 Dart 模型——Provider 自己没帮上忙）。

Riverpod 把注册表整个搬出 Element 树：provider 是全局最终变量（编译期就存在，引用错了直接编译不过），组合用 `ref.watch` 自由声明。代价与收益，S6 你亲手写 v4 时对照体会。

### 练习 1：Selector vs 整页 watch（真实数字）

方法：给列表页每张 ProductCard 临时包 `RebuildBadge('卡片$index')`，用一次性 widget 测试驱动"首帧 → 加购 1 → 加购 2"，读出徽标计数。两种写法各跑一遍（跑完代码已还原）。

| 时点 | 方案 A：`Selector` 角标（现行代码） | 方案 B：`_ListScaffold.build` 顶部 `context.watch` |
|---|---|---|
| 首帧 | 角标:1 · 卡片0:1 · 卡片1:1 | 角标:1 · 卡片0:1 · 卡片1:1 |
| 加购 1 件 | 角标:2 · **卡片0:1 · 卡片1:1** | 角标:2 · **卡片0:2 · 卡片1:2** |
| 再加购 1 件 | 角标:3 · **卡片全程 1** | 角标:3 · **卡片0:3 · 卡片1:3** |

结论：方案 B 里**每加购一件，屏上每张卡片陪跑一次**（20 张可见卡片就是 20 次无效 build，含 Image 子树）；方案 A 里除角标外零陪跑。这两组数字留作 S5 横向对比素材（v0 根 setState 的表现 = 方案 B）。

### 练习 2（思考题）：如果状态必须是不可变对象、每次整体替换？

**换来什么：**

- **"变了什么"变成可计算的**：新旧两个 state 对象可以 diff、可以判等——`buildWhen: (prev, curr) => prev.totalCount != curr.totalCount` 这种"声明式 Selector"才有可能。可变模型里"改哪了"只有方法作者知道,监听者一律全量重查。
- **状态转移可测试、可回放**：每次变更是 `旧state + 事件 → 新state` 的纯函数,单测断言的是值(`expect(states, [A, B, C])`,S3 的 blocTest 就长这样);出 bug 时把状态序列打出来就是"发生了什么"的完整录像(time-travel debugging 的地基)。
- **杜绝一类 bug**：不可变对象不可能被"绕过通知偷偷改"——CartModel 用 `List.unmodifiable` 堵门是运行时防御,不可变状态是类型级根治。

**失去什么：**

- **样板**：每个 state 要 `copyWith`、要值语义 `==`/`hashCode`（Equatable/freezed 就是来减这个负的）;改一个字段要整体复制替换。
- **分配开销**：高频小变更次次新建对象（绝大多数 App 无感,但要知道账在哪）。
- **心智转换**：从"调方法改字段"换成"事件进、状态出"的单向数据流——这正是 S3 Bloc 的世界观,带着这份答案开课。
