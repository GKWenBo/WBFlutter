# S1 自测答案（对照版）

> 对应 [S1-原理篇InheritedWidget与手写迷你Provider.md](../S1-原理篇InheritedWidget与手写迷你Provider.md) 第五节自测清单。

## 1. `of()` 和 `read()` 各做了什么？分别允许在哪些时机调用？为什么 initState 里不能 `dependOn`？

- **`of()`** = `dependOnInheritedWidgetOfExactType`：查表拿到祖先 provider **+ 互相登记依赖**（调用方进 provider 的 `_dependents` 名单）。notifier 开火时名单上的 Element 挨个重建。只能在 **build / didChangeDependencies** 里调。
- **`read()`** = `getInheritedWidgetOfExactType`：只查表取值，**不登记**。initState、事件回调、异步回调里随便调；代价是值变了没人通知你。

initState 里不能 `dependOn`，因为依赖登记必须发生在 Element 的 build 阶段——框架靠"谁在 build 谁登记"保证依赖关系和树结构一致；initState 时首次 build 还没开始，登记没有归属，直接断言拦下。**这就是 S2 里 `context.watch` / `context.read` 分工的框架级原因**，不是 API 设计者的口味。

口诀：展示用 of（要跟新），动手用 read（只办事）。类比 iOS：of ≈ `@EnvironmentObject`（观察），read ≈ 从环境里拿一下对象调个方法（不观察）。

## 2. 为什么每条 push 都要 re-provide？包进去的是新 controller 还是同一个实例？对应 Provider 的哪个 API？

InheritedWidget 的查找表沿 **Element 父链**复制传递；push 出来的路由挂在 Navigator/Overlay 之下，它的父链经过 MaterialApp，**不经过发起 push 的页面**——所以下层页面头顶的 MiniProvider 对新路由不可见，必须在 `MaterialPageRoute.builder` 里再包一层。

包的是**同一个 controller 实例**（先 `read()` 拿到，再塞给新 MiniProvider）——re-provide 的是"可见性"，不是状态本体；五条路由共享同一份购物车。对应 Provider 的 **`ChangeNotifierProvider.value`**（已有实例复用 → `.value`；新建并托管生命周期 → `create`，这是 Provider 的经典面试题）。

另一条标准解法是把 provider 提到 Navigator 之上（包住 MaterialApp），所有路由天然可见——S2 的标准姿势；StateLab 因为五版本共存互不污染才选了 re-provide。

## 3. MiniProvider 的自定义 Element 为什么必须订阅 notifier？InheritedWidget 天生的通知路径是哪条、缺了哪条？

InheritedWidget 天生只有**路径①：widget 实例被替换**——父级重建传了个新的 provider widget，`update()` 里问 `updateShouldNotify(old)`，true 才 `notifyClients`。它是**不可变**的，自己不会"知道"内部数据变了。

缺的是**路径②：数据对象开火 → 依赖者重建**。这条路必须有人去订阅 notifier——就是自定义 `_MiniProviderElement`：构造时 `addListener`，听到通知就置脏 + `markNeedsBuild`，下一帧自己 build 时 `notifyClients` 把消息扩散给依赖名单，unmount 时退订防泄漏。

所有"数据变了自动刷新"的方案都在自己造路径②：Provider 用 ChangeNotifier、Bloc 用 Stream、GetX 用 Rx——零件不同，位置相同。

## 4. `updateShouldNotify` 在日常 `notifyListeners` 刷新时会被调用吗？它到底管什么场景？

**不会。** 这是本课最容易搞混的一点。`notifyListeners` 走的是路径②（Element 订阅 → notifyClients），全程不碰 `updateShouldNotify`。

它只管路径①：**provider widget 实例被替换**时（父级 setState 重建、热重载），新旧 widget 对比"值得不值得惊动依赖者"。MiniProvider 里写的是 `notifier != oldWidget.notifier`——只有换了一个 notifier 实例才通知；同一个实例只是外壳重建，依赖者不用陪跑。

## 5. 购物车页为什么能从 Stateful 变 Stateless？"假 Stateful"是什么代码味？

购物车页从头到尾**没有一个属于自己的状态**——展示的数据（items、合计）全是共享购物车的投影。S0 时它被迫 Stateful，唯一原因是需要 `setState(() {})` 来通知**自己**重建（双重 setState 的"本页那次"）。现在通知由 MiniProvider 的依赖机制接管：页面 build 里 `of()` 登记依赖，谁改购物车本页自动重建——State 失业，降级 Stateless。

**"假 Stateful"代码味**：State 类里没有自己的字段（或只有转发用的 getter），setState 调用全是空闭包 `setState(() {})`。看到它就该问：这个页面缺的不是状态，是**通知机制**——这是裸共享数据的典型症状。类比 iOS：一个 VC 里全是 `tableView.reloadData()` 却没有一行自己的属性，说明数据和通知的归属错位了。

## 6. 依赖（重建）的粒度由什么决定？列表页是怎么让商品卡不陪跑的？S2/S3/S4 各自对应的收窄工具叫什么？

粒度 = **调 `of()`（dependOn）的那个 Element**。登记进名单的是 Element，不是"页面"——provider 通知时只重建名单上的人。

列表页的做法：页面 State 的 build 里**不调** of()，把它推进 AppBar 里的一个小 `Builder`。于是依赖者只有那个 Builder（角标块）——加购时角标 RebuildBadge +1，整页和所有商品卡纹丝不动。对照 S0：根 setState 让整页 + 全部卡片陪跑。

同一思想的后续工具：S2 Provider 的 **`Selector` / `context.select`**（依赖到"字段"级）、S3 Bloc 的 **`buildWhen`**（新旧 state 对比决定要不要重建）、S4 GetX 的 **`Obx`**（自动收集 Rx 依赖，粒度就是 Obx 块）。全是"名单越小、重建越少"。

## 7. CartController 为什么要用 `List.unmodifiable` 交出数据？它封死了 v0 的哪个病灶？

v0 的根病灶：**裸共享可变 List**——谁都能 `cart.clear()`，但没有任何机制保证改完有人发通知，"改了没人知道"就是你抓到的角标陈旧 bug 的本质。

`List.unmodifiable` 让外部拿到的永远是只读视图：绕过方法直接改，运行时立刻 `UnsupportedError`（有单测钉住）。于是"改数据"只剩一条路——走 controller 的方法，而方法末尾必有 `notifyListeners()`。**改和通知被锁进同一扇门，"只改不通知"从结构上不可能发生。**代价是每次 getter 都拷贝一份，购物车这个量级可以忽略；高频大列表要换只读包装（如 `UnmodifiableListView`，O(1) 视图不拷贝）。

## 8. controller 谁创建、谁 dispose？忘了 dispose 会怎样？

**谁创建，谁负责销毁**：`_V0ShopRootState` 在字段里创建 `CartController`，就在自己的 `dispose()` 里调 `_cart.dispose()`（≈ iOS 根 VC 持有 viewModel，跟着 deinit 一起走）。MiniProvider 只是"挂树"，不管生命周期——这正是它比 Provider 裸的地方：`ChangeNotifierProvider(create: ...)` 会替你托管 create + dispose，S2 对照。

忘了 dispose 的后果：① controller 的监听者名单一直持有各 Element 的回调，对象泄漏；② 如果还有悬挂的异步回调之后调 `notifyListeners`，可能撞上 "A CartController was used after being disposed" 或幽灵刷新。ChangeNotifier 的 debug 断言能抓一部分,但别指望它兜底。

## 9. v0 重构后还剩哪些 setState？为什么它们应该留下？

剩下的全是**页面私有状态**（场景①②⑤）：

- 列表页：`_loading/_loadingMore/_error/_items/_hasMore/_skip`（异步三态 + 分页）；
- 详情页：`_favorite`（收藏心形，页面销毁即消失）；
- 搜索页：`_results/_loading/_error/_lastQuery`（输入流处理）。

它们该留下，因为**没有第二个页面关心这些值**——不跨页、不共享、生命周期与页面同长。给它们上全局状态管理是负优化：多一层样板、多一份心智负担、还把"页面私有"这个语义弄丢了。**判断标准从来不是"用不用 setState"，而是"这份状态几个人看"**：一个人看 → setState 舒适区；多人看 → 进 controller 上树。这条判断标准五个版本通用（Riverpod 时代收藏心形也照样是本地 State）。

## 10. `Theme.of(context)` 和本课机制什么关系？

同一套机制。`Theme.of(context)` 内部就是 `context.dependOnInheritedWidgetOfExactType<InheritedTheme>` ——查表 + 登记依赖；换肤时 Theme 被替换，`updateShouldNotify` 判真，所有调过 `Theme.of` 的 Widget 自动重建（走的是路径①）。`MediaQuery.of`、`Localizations.of`、`Navigator.of`（这个用的是不登记的变体）全是一家人。

所以本课学的不是某个冷门 API，而是你**每天都在用**的 Flutter 骨架:MiniShop 页面里每一行 `Theme.of(context)`，和你手写的 `MiniProvider.of<CartController>(context)`,在 Element 眼里是同一件事。旋转屏幕时页面自动重排（MediaQuery 变化）——那就是路径①在生产环境里的日常演出。
