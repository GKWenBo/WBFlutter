# StateLab 状态管理专题课程设计

> 2026-07-11 定稿。本文档是 StateLab 学习线的**唯一权威设计文档**，后续每课的实施计划、课时讲义、技术文档都以此为准。修改设计先改这里。

## 1. 背景与目标

学员是资深 iOS 工程师（第三条学习线，前两条：WanShop 电商实战 M0–M13、NativeLab 原生交互 L0+）。已掌握 `setState` 与 Riverpod 的日常用法，本专题的目标：

1. **补全方案版图**：系统掌握 Provider、Bloc、GetX 三个主流方案（大量存量项目与面试高频考点）。
2. **打穿原理层**：理解所有方案共同的地基——InheritedWidget 依赖注册、Listenable 通知机制——做到"知其所以然"，而不是会调 API。
3. **建立选型判断力**：同一个需求五种写法（含 setState 基线与 Riverpod 结课作业），横向对比重建范围、样板量、可测性、心智模型，形成自己的选型决策树。

**核心交付物**（学员点名要求）：每种方案一篇**深度技术文档**，沉淀在 `docs/tech/`。

## 2. 已确认决策（不要再问）

| 决策点 | 结论 |

| Riverpod | 纳入对照，但作为**学员结课作业**（S6）独立完成，讲师只出验收 checklist 和 code review |

## 3. 工程结构

```
state_lab/                        # 与 native_lab 平级的第三条学习线
  lib/
    main.dart                     # 极薄入口，只 runApp（沿用 WanShop 约定）
    app/                          # StateLabApp + 首页：方案列表页（对齐 native_lab 课程列表）
    shared/                       # 共享底座——只写一遍，各方案版本复用
      models/                     #   Product / CartItem 等 DummyJSON 建模
      api/                        #   dio 客户端 + ProductApi（分页、搜索、详情）
      widgets/                    #   纯展示组件：ProductCard、三态视图骨架、
                                  #   购物车角标、RebuildBadge 重建计数器
    versions/                     # ⭐ 五份 MiniShop 实现，规格完全一致
      v0_setstate/                # S0 基线版：setState + 构造函数传参
                                  #（S1 会就地小重构：手写迷你 Provider 替换购物车层层传参，
                                  #  痛点原版随时可从 git 历史回看）
      v1_provider/                # S2
      v2_bloc/                    # S3
      v3_getx/                    # S4
      v4_riverpod/                # S6 结课作业（学员写）
  test/                           # 每课的状态层单测 + widget 测试
  docs/
    2026-07-11-状态管理专题课程设计.md   # 本文档
    plans/                        # 一课一实施计划（对齐 NativeLab 惯例）
    lessons/                      # 课时讲义 SX-*.md + README.md 进度表 + 自测答案/
    tech/                         # ⭐ 深度技术文档（核心交付物）
      s1-状态管理的地基.md
      s2-provider.md
      s3-bloc.md
      s4-getx.md
      s5-横向对比与选型.md
```

**共享底座原则**：数据模型、网络层、纯展示 Widget 与状态管理无关，只写一遍。每个 `versions/vX_*/` 只包含**状态层 + 页面装配**。这样任何两份实现的 diff 就是纯粹的状态管理差异——对比信号最强，噪音最少。共享 Widget 必须保持"纯展示"（只收数据和回调，不 import 任何状态管理库），这是硬约束。

**两类文档分工**（避免互相复制）：

- **课时讲义** `docs/lessons/SX-*.md`：管上课节奏——本课重点、代码地图、新控件/API 速查表（含 iOS 类比）、双端对照、自测清单、课后练习。薄，引用技术文档不复制。
- **技术文档** `docs/tech/*.md`：每方案一篇深度长文，统一章节骨架：
  1. 心智模型与 iOS 类比（Provider≈EnvironmentObject、Bloc≈Combine 单向数据流、GetX≈RxSwift）
  2. API 全景速查表
  3. MiniShop 实战导读（指向 versions/ 代码）
  4. 底层原理（如何触发 rebuild、依赖收集机制、与 InheritedWidget 的关系）
  5. 优缺点、适用场景、常见坑
  6. 面试高频题
- 工程内其余惯例照旧：`app/` 首页带进度门禁锁（完成一课解锁下一课），仅 iOS/Android 双平台，org `com.wenbo`。

## 4. MiniShop 需求冻结

四个页面，规格冻结，五份实现必须完全一致：

| 页面 | 功能 | 覆盖的状态场景 |
|---|---|---|
| 商品列表 | DummyJSON 分页加载（每页 20 条）、下拉刷新、触底加载更多 | ① 异步三态（loading/error/data）+ 分页追加 |
| 商品详情 | 大图 + 信息、收藏心形切换、加入购物车 | ② 局部 UI 状态（收藏只活在本页，不入全局） |
| 购物车 | 数量增减、滑动删除、合计金额/件数、清空 | ③ 跨页共享 ④ 派生状态（总价/总数从 items 算出） |
| 搜索 | 输入防抖 400ms、即时结果、丢弃过期请求 | ⑤ 输入流处理 |

全局设计两条：

- **AppBar 购物车角标全页面常驻**，数量实时变化。在列表页加购、角标立刻 +1——这是观察每种方案"谁在监听、谁被重建"的最佳窗口。
- **RebuildBadge 重建计数器**（debug 小徽标，放 shared/widgets/）：挂在关键区块上显示各自 build 次数。对比 Provider 的 `Selector`、Bloc 的 `buildWhen`、GetX 的 `Obx` 如何收窄重建范围时，肉眼可见。

这 5 个场景同时是 S5 横向对比文档的骨架：同一场景，五种方案代码并排。

## 5. 课时安排（S0–S6）

| 课 | 主题 | 内容与产出 |
|---|---|---|
| S0 | 工程起步 + setState 基线 | 创建工程、首页方案列表、shared 底座（models/api/widgets）、**v0_setstate 完整 MiniShop**。故意用最朴素写法：购物车靠构造函数层层下传、回调层层上浮——亲手体验痛点，后面每个方案讲"它解决了 S0 的什么"。讲义 S0。 |
| S1 | 原理篇：所有方案的地基 | InheritedWidget 的依赖注册与 `updateShouldNotify`、Element 树上的查找路径、ChangeNotifier/ValueNotifier/ListenableBuilder。**手写 ~40 行迷你 Provider**（InheritedNotifier 思路）替换 v0 的购物车层层传参，v0 就地小重构。技术文档《s1-状态管理的地基》。 |
| S2 | Provider | **v1_provider 完整实现**。ChangeNotifierProvider / Consumer / Selector / context.watch·read·select / MultiProvider / ProxyProvider；与 S1 手写版对照"Provider 到底帮你做了什么"。技术文档 s2。 |
| S3 | Bloc | **v2_bloc 完整实现**。先 Cubit（购物车，轻量入门）后 Bloc 事件驱动（商品列表：Fetch/LoadMore/Refresh 事件流）；BlocBuilder / BlocListener / BlocConsumer / buildWhen；Equatable 与状态不可变；搜索防抖用 EventTransformer。`blocTest` 单测。技术文档 s3。 |
| S4 | GetX | **v3_getx 完整实现**。GetBuilder 手动挡 / Obx+Rx 自动挡双轨；GetxController 生命周期；Get.put/find/lazyPut 依赖注入 + Bindings；workers 做搜索防抖。**专节讲全家桶（路由/DI/工具）的价值与风险边界、为什么社区有争议**——课内路由体验 Get.to，但讲清楚不依赖 context 的代价。技术文档 s4。 |
| S5 | 横向对比总结 | 代码量少、文档为主：五方案同场景代码并排、心智模型对比图、重建范围实测（用 RebuildBadge 数据）、样板量/可测性/学习曲线/团队协作对比表、**选型决策树**、迁移成本、面试题汇总（20+ 道带答案要点）。技术文档 s5。 |
| S6 | 结课作业：Riverpod | 学员独立完成 **v4_riverpod**（Notifier/AsyncNotifier + codegen，与 WanShop 同栈但场景不同）。讲师产出：验收 checklist（对照第 4 节场景清单 + 重建范围要求）→ 学员提交后做 code review → 通过即结课。 |

## 6. 流程与验收（沿用既有两条学习线的规矩）

- **每课节奏**：概念讲解 → 写代码 → `flutter analyze` 0 issue → iPhone 17 模拟器（`85C08A29-994F-44BC-8BEF-C0CB6D6DBFF7`）截图验证 → 学员确认过关 → git 提交 → 才开下一课。
- **课时讲义必须在发出"请你验证"消息之前写好**（硬性时序，见教学约定）；自测答案归 `docs/lessons/自测答案/`。
- **每课配核心单测**：状态层是最易测的层——Provider/GetX 直接实例化 controller 断言，Bloc 用 `blocTest`，加少量 widget 测试。全量 `flutter test` 必须绿。
- **一课一实施计划**：开课前先写 `docs/plans/YYYY-MM-DD-SX-*.md`，学员说"开始"再动工（对齐 NativeLab 模式）。
- **进度真身**：`docs/lessons/README.md` 进度表，一课一行。
- **git**：只 add `state_lab/`，中文提交信息 `SX 主题：…`，提交前等学员确认。
- **依赖版本**（S0 创建时锁定，2026-07-11）：`dio ^5.10.0` · `provider ^6.1.5+1` · `flutter_bloc ^9.1.1` · `equatable ^2.1.0` · `get ^4.7.3` · `flutter_riverpod ^3.3.2` · `json_annotation ^4.12.0` · dev: `mocktail ^1.0.5` · `json_serializable ^6.14.0` · `build_runner ^2.15.1`。Flutter 3.44.4 stable。JSON 解析统一走 json_serializable 代码生成（2026-07-12 学员定的规矩：不手写 fromJson）。riverpod 的 codegen 工具链（riverpod_annotation / riverpod_generator / build_runner）S6 由学员自行添加。

## 7. 不做什么（YAGNI）

- 不覆盖 MobX、signals、redux 等小众方案（S5 对比文档可一句话提及定位）。
- 不做登录/下单/支付——MiniShop 只为状态场景服务，WanShop 已覆盖完整电商业务。
- 不做国际化、不做 golden 测试、不追求 UI 精美——共享 Widget 够用即可。
- GetX 路由/全家桶只在 S4 讲边界，主工程导航不绑定 GetX。
- 不写"第六种方案"的完整实现；学员未来想加（如 signals）可按 versions/ 模式自行扩展。
