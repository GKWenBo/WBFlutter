# s1 · 状态管理的地基：InheritedWidget 与 Listenable

> StateLab 技术文档第一篇（S1 课配套）。后面 s2-provider / s3-bloc / s4-getx 的"底层原理"章都会回链到这里。
> 实战代码：`lib/versions/v0_setstate/state/`（CartController + MiniProvider）。

## 0. 一句话总纲

**Flutter 的一切状态管理方案，都是对两个问题的不同回答：**

1. **DI 问题**：状态对象放哪儿，子孙 Widget 怎么拿到它？——答案的地基是 `InheritedWidget`（沿树注入、按类型查找）。
2. **订阅问题**：状态变了，通知谁重建？——答案的地基是 `Listenable`（`ChangeNotifier` / `ValueNotifier` 的监听机制）。

Provider = 这两个地基的官方封装；Bloc 把"订阅"换成 Stream；GetX 把"查找"换成全局注册表、把"订阅"换成 Rx；Riverpod 干脆把"树上查找"整个搬出了 Widget 树。**地基打穿，后面每个方案都只是在回答"它俩换了什么零件"。**

## 1. 心智模型与 iOS 类比

| Flutter | iOS 对应物 | 说明 |
|---|---|---|
| `InheritedWidget` | SwiftUI `Environment` / `environmentObject` | 挂在树上、沿树向下可见、按**类型**查找 |
| `dependOnInheritedWidgetOfExactType` | `@EnvironmentObject` | 取值 **+ 自动订阅**（值变了我重建） |
| `getInheritedWidgetOfExactType` | 拿一下对象但不观察 | 取值**不订阅** |
| `ChangeNotifier` | `ObservableObject` | `notifyListeners()` ≈ `objectWillChange.send()` |
| `ValueNotifier<T>` | 单个 `@Published` 属性 | 只装一个值，值变即通知（用 `==` 判断） |
| `ListenableBuilder` | Combine `sink` + 手动刷新那一小块 | 不经过树上查找，直接对某个 Listenable 建订阅点 |
| `InheritedNotifier` | 上面两行的胶水 | 我们手写的 MiniProvider 就是它 |

SwiftUI 里 `@EnvironmentObject var cart: Cart` 一行背后发生的两件事——从环境里按类型找到实例、把本 View 登记为它的观察者——在 Flutter 里被拆开成看得见摸得着的两个 API。这是 Flutter"显式到啰嗦、但没有魔法"的典型风格。

## 2. API 全景速查表

| API | 用途 | 关键点 | 坑 |
|---|---|---|---|
| `InheritedWidget` | 沿树注入数据 | 子类持 final 数据 + 实现 `updateShouldNotify` | 它自己不可变；"数据变了通知"需要配 Listenable（见 InheritedNotifier） |
| `updateShouldNotify(old)` | 新旧 widget 替换时，要不要通知依赖者 | 只在**provider widget 实例被替换**时调用（父级重建/热重载） | 日常 `notifyListeners` 刷新**不走它**——最常被误解的一点 |
| `context.dependOnInheritedWidgetOfExactType<T>()` | 查找 + 注册依赖 | O(1) 查表；调用方 Element 进依赖名单 | 只能在 `build`/`didChangeDependencies` 里调；`initState` 里禁用 |
| `context.getInheritedWidgetOfExactType<T>()` | 只查找不注册 | 事件回调/`initState` 可用 | 值变了不会通知你——别拿它做展示数据源 |
| `InheritedNotifier<T extends Listenable>` | InheritedWidget + Listenable 胶水 | notifier 开火 → 自动通知依赖者 | 本课手写了它的等价物，见 §4.3 |
| `InheritedModel<T>` | 带 aspect 的 InheritedWidget | 依赖时声明"我只关心哪一面"，通知按面过滤 | 小众；知道定位即可（Provider 的 `select` 是更好用的替代） |
| `ChangeNotifier` | 可监听对象基类 | `addListener/removeListener/notifyListeners/dispose` | 忘 `dispose` 泄漏；同步通知，别在 build 期间开火 |
| `ValueNotifier<T>` | 单值 ChangeNotifier | `value` setter 里 `==` 变了才通知 | 装集合时改内容不换引用 == 不通知（`list.add` 后 value 没变） |
| `ListenableBuilder` | 对 Listenable 建局部订阅点 | `listenable:` + `builder:`，不依赖树上查找 | 3.10+ 才有；前身是 `AnimatedBuilder`（同一个类的别名关系） |
| `ValueListenableBuilder<T>` | 对 ValueNotifier 建订阅点 | builder 直接给 `value`，附带 `child` 缓存 | 记得用 `child` 参数缓存不变的大子树 |

## 3. MiniShop 实战导读（S1 重构）

对照命令：`git diff 93abcd2 HEAD -- state_lab/lib/versions/v0_setstate/`（93abcd2 = 重构前最后一个纯 v0 提交）。

### 3.1 CartController：改与通知锁进同一扇门

v0 的根病灶不是 setState 本身，而是**裸共享**：谁都能改那个 List，但没有任何机制保证"改了有人通知界面"。S0 你抓到的 pop 返回角标不清零 bug，本质就是"改了、但没人通知详情页"。

`CartController extends ChangeNotifier` 的三个设计决定：

1. **`items` 只暴露 `List.unmodifiable` 视图**——外部拿不到可变引用，想改必须走方法（编译器帮你守门，`cart.items.clear()` 直接抛 `UnsupportedError`，有单测）。
2. **每个变更方法末尾 `notifyListeners()`**——"改"和"通知"在同一个函数体里，物理上不可能只改不通知。
3. **派生值仍是 getter 现算**（totalCount/totalPrice）——场景④原则从 S0 一路带到 S6。

### 3.2 MiniProvider：40 行打通两个地基

用法三件套：

```dart
MiniProvider(notifier: cart, child: page)          // 挂树（≈ .environmentObject）
MiniProvider.of<CartController>(context)           // 取值+订阅（≈ @EnvironmentObject）
MiniProvider.read<CartController>(context)         // 只取值（事件回调用）
```

三处关键 diff 读法：

- **根**（`v0_shop_root.dart`）：四个转发方法删光，根只剩"创建 controller、dispose controller、挂 MiniProvider"三件事。
- **列表页**：构造函数 6 参 → 1 参；`of()` 收进 AppBar 里的一个 `Builder`——**依赖粒度 = 调 of() 的那个 Element**，购物车变化时只有角标块重建，商品卡不陪跑（RebuildBadge 计数可证）。
- **购物车页**：`StatefulWidget → StatelessWidget`。它本来就没有自己的状态，v0 时代被迫 Stateful 只为了能 setState 通知自己——这类"假 Stateful"是识别裸共享代码味的信号。

### 3.3 为什么每条 push 都要再包一层

`_openDetail` 里的固定句式：

```dart
final cart = MiniProvider.read<CartController>(context);
Navigator.of(context).push(MaterialPageRoute<void>(
  builder: (_) => MiniProvider(notifier: cart, child: V0ProductDetailPage(product: product)),
));
```

原因见 §4.4（InheritedWidget 不跨路由）。注意包的是**同一个 controller 实例**——S2 会看到 `ChangeNotifierProvider.value` 干的就是这件事,连"什么时候该用 .value 而不是 create"这个 Provider 经典面试题的答案都藏在这里（已有实例复用 → value；新建并托管生命周期 → create）。

## 4. 底层原理

### 4.1 查找为什么是 O(1)：`_inheritedElements` 表

`dependOnInheritedWidgetOfExactType` 听起来像"沿着父链一层层爬直到找到"，实际不是。每个 Element 身上带一张哈希表 `_inheritedElements`（`Type → InheritedElement`，实现是持久化哈希映射 `PersistentHashMap`）：

- 普通 Element 挂载（mount）时，直接**继承**父 Element 的表引用；
- `InheritedElement` 挂载时，**复制**父表并把自己按类型写进去，再传给子树。

于是任何深度的 Widget 查找任何 InheritedWidget 都是一次哈希查表。代价换到了挂载期：这也是"InheritedWidget 越靠近根、覆盖越广、成本越均摊"的原因。

> 对 iOS 的映射：SwiftUI 的 EnvironmentValues 也是沿视图树复制传递的一包值，思路同源。

### 4.2 依赖注册：一张名单

`dependOnInheritedWidgetOfExactType<MiniProvider<CartController>>()` 做两件事：

1. 查表拿到那个 `InheritedElement`；
2. 互相登记：调用方 Element 把 provider 记进自己的 `_dependencies`，provider element 把调用方记进自己的 `_dependents` 名单。

之后 provider 需要广播时（`notifyClients`），就是**遍历名单挨个调 `dependent.didChangeDependencies()`**，后者内部 `markNeedsBuild()`——下一帧这些依赖者重建。依赖者 unmount 时自动从名单除名，不需要手动"反注册"。

**依赖的粒度是 Element，不是页面**。这就是列表页把 `of()` 收进小 `Builder` 就能收窄重建范围的原理——也是 S2 `Selector`、S3 `buildWhen`、S4 `Obx` 全部优化手段的共同祖先：**谁调的 watch，谁上名单；名单越小，重建越少。**

### 4.3 通知的两条路径（最容易混淆处）

**路径①：provider widget 实例被替换**（父级重建传了新对象/热重载）：
`Element.update(newWidget)` → `updated()` 里问 `updateShouldNotify(old)` → true 则 `notifyClients`。

**路径②：notifier 开火**（日常刷新走这条）：
`notifyListeners()` → 自定义 element 的 `_handleUpdate`（它在挂载时 `addListener` 订阅了 notifier）→ `_dirty = true; markNeedsBuild()` → 下一帧该 element `build()` 时 `notifyClients` → 依赖者们 `didChangeDependencies` → 重建。

MiniProvider 手写的就是路径②，与 framework 的 `inherited_notifier.dart` 逐行同构（打开源码对照：`_InheritedNotifierElement` 的 constructor 订阅、`update` 换订阅、`_handleUpdate` 置脏、`build` 里 `notifyClients`、`unmount` 退订——五个动作一一对应）。**InheritedWidget 天生只有路径①**；所有"数据变了自动刷新"的方案，都要自己造路径②——Provider 用 ChangeNotifier，Bloc 用 Stream，GetX 用 Rx，形式不同，位置相同。

### 4.4 为什么不跨路由（你那个 bug 的完整答案）

`_inheritedElements` 表是**沿 Element 父链**传下来的。`Navigator.push` 出来的新路由，Element 挂在 `Navigator`/`Overlay` 之下——它的父链经过 Navigator、MaterialApp，**不经过你发起 push 的那个页面**。所以：

- v0 时代：详情页读得到 cart（构造函数递过去的引用），但根的 setState 够不着它 → 陈旧 UI；
- S1 时代：详情页如果不 re-provide，`of()` 会直接断言失败（"这条路由包了吗？"）——**从"默默陈旧"变成"当场报错"，这本身就是进步**。

两种标准解法：

1. **provider 提到 Navigator 之上**（包住 MaterialApp）：所有路由天然可见。Provider 官方推荐姿势，S2 采用；代价是状态生命周期 = App 生命周期。
2. **每条路由 re-provide 同一实例**（本课姿势 ≈ `Provider.value`）：状态生命周期仍归属版本子树，但每条 push 要多包一行。

StateLab 因为"五版本共存、互不污染"选了 2；单方案 App 通常选 1。

### 4.5 为什么 `of()` 不能在 initState 里调

依赖注册要求调用方 Element 处于 build 阶段（框架靠这个保证依赖关系与树结构一致）。`initState` 时 Element 还没进入首次 build，`dependOn` 会断言失败。规则：

- **build / didChangeDependencies** 里 → `of()`（watch 语义）
- **initState / 事件回调 / 异步回调** 里 → `read()`（get 语义）

这正是 Provider 把 API 拆成 `context.watch` / `context.read` 的框架级原因，不是风格洁癖。

## 5. 优缺点、适用场景、常见坑

**裸用 InheritedWidget/InheritedNotifier 的定位**：理解成本最低的"地基层"，但样板不少——要自己管 controller 生命周期（Stateful 宿主）、自己处理跨路由、没有 select 细粒度。**Provider 本质上就是把这些样板卖给你**：`ChangeNotifierProvider` = Stateful 宿主 + create/dispose 托管；`Consumer/Selector` = 圈依赖粒度的 Builder 的强化版。S2 逐一对照。

常见坑清单：

1. `of()` 忘 re-provide 就 push → 断言"这条路由包了吗"（好事，当场炸）。
2. 在 `initState` 里 `dependOn` → 断言失败（§4.5）。
3. controller 忘 `dispose` → 泄漏 + 潜在"dispose 后 notifyListeners"崩溃。
4. `updateShouldNotify` 无脑 `true` + 父级频繁重建 → 通知风暴，依赖者全员陪跑。
5. `ValueNotifier<List>` 改内容不换引用 → `==` 没变、不通知（要么换新 List，要么用 ChangeNotifier 手动 notify）。
6. 在 build 期间同步 `notifyListeners` → "setState during build" 同款异常。

## 6. 面试高频题（答案要点）

1. **InheritedWidget 如何做到 O(1) 查找？** 每个 Element 携带 `_inheritedElements` 哈希表（Type→InheritedElement），InheritedElement 挂载时复制父表写入自己再下传；dependOn 是查表不是爬树。
2. **`updateShouldNotify` 什么时候被调用？`notifyListeners` 触发的刷新走它吗？** 只在 provider widget 实例被替换（父级重建/热重载）时；不走——那是 InheritedNotifier 类 element 的监听路径（两条通知路径,见 §4.3）。
3. **`didChangeDependencies` 和 `build` 的关系？** 依赖的 InheritedWidget 变化时被调，默认实现会走到 markNeedsBuild → 随后 build；重逻辑（如依赖变化才需要重新拉数据）放这里，纯展示放 build。
4. **Provider 和 InheritedWidget 什么关系？** Provider = InheritedWidget（查找/依赖）+ ChangeNotifier（通知）+ 生命周期托管（create/dispose）+ 细粒度 API（Selector/select）的封装；没有绕开框架的私有魔法。
5. **`context.watch` 和 `context.read` 的区别？各自能在哪调？** watch = dependOn（订阅，build/didChangeDependencies 限定）；read = getInherited...（不订阅，initState/回调随便调）。在 build 里滥用 read 会拿到旧值且不跟新。
6. **为什么 push 出去的页面拿不到下层 provider？** 查找表沿 Element 父链传递；新路由父链挂在 Navigator/Overlay 下，不经过发起页。解法：provider 提到 Navigator 之上，或 push 时 re-provide 同一实例（.value）。
7. **ChangeNotifier 和 ValueNotifier 怎么选？** 多字段/多方法的领域状态 → ChangeNotifier；单值且值可用 `==` 判等 → ValueNotifier（省手动 notify）。集合慎用 ValueNotifier（引用不变不通知）。
8. **InheritedModel 解决什么？** 依赖时声明 aspect，通知按 aspect 过滤——"我只依赖这份数据的某一面"。生产中更常用 Provider 的 `select` 达成同效。
9. **手写一个"数据变了自动刷新子树"的最小实现思路？** InheritedWidget 存 Listenable + 自定义 InheritedElement：constructor 订阅、update 换订阅、听到通知置脏 markNeedsBuild、build 时 notifyClients、unmount 退订。（≈ 本课 MiniProvider / framework InheritedNotifier。）
10. **`Theme.of(context)` 为什么能让换肤自动生效？** Theme 内部就是 InheritedWidget（InheritedTheme）；of() 用的同一套 dependOn 依赖注册——你天天在用的 MediaQuery/Theme/Localizations 全是这个机制。
